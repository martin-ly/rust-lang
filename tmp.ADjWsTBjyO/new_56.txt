### 5.6 补充：`Vec<T>` / `String` / `HashMap` 的内存布局与扩容策略

> **[Rust Reference: Vec]** · **[Rust Reference: String]** · **[std::collections::HashMap]** · **[Wikipedia: Dynamic array]** · **[Abseil: SwissTable]** 理解标准库容器的内存布局是系统编程的核心能力。Rust 的标准库容器在设计上追求**缓存友好**与**摊还 O(1) 性能**的平衡。✅

#### `Vec<T>`：三字段连续内存与指数扩容

```rust
use std::mem;

// ✅ Vec 内存布局（std 实际定义的逻辑视图）
// struct Vec<T> {
//     buf: RawVec<T>,  // 封装 ptr + capacity
//     len: usize,
// }
// 等价于逻辑视图: { ptr: *mut T, len: usize, cap: usize }

// 64 位系统下 Vec<i32> 的元数据大小 = 24 字节（3 × usize）
assert_eq!(mem::size_of::<Vec<i32>>(), 24);
assert_eq!(mem::size_of::<Vec<u64>>(), 24);

// ✅ 预分配避免频繁扩容
let mut v = Vec::with_capacity(4);
assert_eq!(v.len(), 0);
assert_eq!(v.capacity(), 4);
v.push(1); v.push(2); v.push(3); v.push(4);
// 第 5 个元素触发扩容
v.push(5);
assert!(v.capacity() >= 8);  // 通常翻倍
```

| 操作 | 时间复杂度 | 触发条件 | 内存行为 |
|:---|:---|:---|:---|
| `push` | 摊还 O(1) | `len < capacity` | 在 `ptr.add(len)` 写入，len += 1 |
| `push`（扩容）| O(n) | `len == capacity` | 分配新内存（通常 2×），move 所有元素，释放旧内存 |
| `pop` | O(1) | `len > 0` | len -= 1，不立即释放内存 |
| `shrink_to_fit` | O(n) | 显式调用 | 重新分配到 `len == capacity` 的精确大小 |
| `reserve` | O(n) | 显式调用 | 预分配至指定 capacity |

**扩容因子与摊还分析**: Rust `Vec` 的扩容因子历史上为 **2×**，现代实现趋向更保守的 **1.5–2×** 区间（具体取决于分配器反馈）。这保证了连续 `push` 的摊还时间复杂度为 O(1)：

```text
假设从 capacity = 1 开始，连续 push n 个元素：
总复制次数 = 1 + 2 + 4 + ... + n/2 + n ≈ 2n - 1
摊还每次 push 成本 = O(2n)/n = O(1)
```

**边界与反例**:

```rust
// ❌ 反例: 频繁扩容导致性能退化
fn slow_push() -> Vec<i32> {
    let mut v = vec![];  // capacity = 0
    for i in 0..10000 {
        v.push(i);  // 频繁触发 O(n) 扩容复制
    }
    v
}

// ✅ 修正: 预分配 capacity
fn fast_push() -> Vec<i32> {
    let mut v = Vec::with_capacity(10000);
    for i in 0..10000 {
        v.push(i);  // 无扩容，始终 O(1)
    }
    v
}

// ❌ 反例: 误用 len() 当字符数（String 同理）
let s = String::from("你好");
assert_eq!(s.len(), 6);   // 字节数 = 6（UTF-8 编码）
// assert_eq!(s.len(), 2); // 错误: len() 不是字符数
```

#### `String`：UTF-8 不变性与 `Vec<u8>` 的特化

```rust
// ✅ String 本质上是 Vec<u8> 的包装，附加 UTF-8 有效性不变式
// struct String {
//     vec: Vec<u8>,
// }

assert_eq!(mem::size_of::<String>(), 24);  // 同 Vec<u8>
```

| 特性 | `String` | `Vec<u8>` |
|:---|:---|:---|
| 内存布局 | `{ ptr: *mut u8, len: usize, cap: usize }` | 同左 |
| 有效性约束 | 必须是合法 UTF-8 字节序列 | 任意字节 |
| `push` | 追加一个 Unicode scalar value（1–4 字节） | 追加单个字节 |
| `capacity` | 按字节计 | 按字节计 |
| 索引 | ❌ 不支持按字符索引（O(n) 风险） | ✅ 支持按字节索引 |

> **关键洞察**: `String` 不是"字符数组"，而是**合法 UTF-8 字节序列**。`String::len()` 返回字节数而非字符数，因为 Unicode scalar value 长度可变（1–4 字节）。这与 Java/C# 的 `String`（UTF-16）形成对比。

**UTF-8 不变性的边界**:

```rust
// ✅ 安全构造: from_utf8 返回 Result，无效时拒绝
let bytes = vec![0x80, 0x81];  // 无效 UTF-8 起始字节
let s = String::from_utf8(bytes);
assert!(s.is_err());  // Err: invalid utf-8 sequence

// ✅ 有损转换: from_utf8_lossy 用 � (U+FFFD) 替换无效序列
let bytes = b"hello \xF0\x90\x80 invalid";
let s = String::from_utf8_lossy(bytes);
assert!(s.contains('�'));  // 替换字符

// ❌ 反例: unsafe 绕过 UTF-8 检查导致未定义行为
// let s = unsafe { String::from_utf8_unchecked(vec![0x80]) };
// UB: 后续假设 String 为合法 UTF-8 的操作可能崩溃
```

#### `HashMap<K, V>`：SwissTable（Robin Hood 哈希 + SIMD 查找）

> **[Abseil: SwissTable]** · **[Rust hashbrown crate]** Rust 的 `HashMap`（基于 `hashbrown`）实现了 **SwissTable** 算法，源自 Google Abseil 的 C++ 实现。其核心是**Robin Hood 哈希**与**SIMD 并行探测**的结合。✅

```rust
// ✅ HashMap 内存布局（Rust std，2024+ 实现）
// struct HashMap<K, V> {
//     base: RawTable<(K, V)>,   // 控制字节数组 + 键值对数组
//     hash_builder: DefaultHashBuilder,  // SipHash 1-3
// }
```

**SwissTable 的三层结构**:

1. **控制字节（control bytes）**: 每个 slot 对应一个 1 字节的元数据：
   - `0b0XXXXXXX`（最高位为 0）: 空槽位，低 7 位为 hash 的 H1 部分
   - `0b11111111`（-1）: 空槽位（tombstone 后的真正空位）
   - `0b10000000`（-128）: 已删除（tombstone）

2. **Robin Hood 哈希**: 当插入时发现已有元素离其"理想位置"更近（probe distance 更短），则**交换**两者位置。这使得所有元素的 probe distance 保持较小且方差低。

3. **SIMD 查找**: 在 x86-64 上利用 SSE2 / AVX2 一次比较 16 个控制字节，快速定位可能的匹配槽位；仅在 SIMD 命中后才进行完整的键相等比较。

| 特性 | Rust HashMap (SwissTable) | C++ `std::unordered_map` |
|:---|:---|:---|
| **冲突解决** | Robin Hood 哈希 + SIMD 线性探测 | 链地址法（bucket + 链表/树） |
| **内存布局** | 单一连续数组（控制字节 + 键值对） | 离散节点分配（指针跳转） |
| **缓存友好性** | 极高（控制字节与数据局部性共存） | 低（指针跳转，缓存 miss 高） |
| **扩容触发** | 负载因子 > 0.875（7/8） | 负载因子 > 1.0（通常） |
| **默认 hasher** | SipHash 1-3（抗 HashDoS） | 通常不抗 HashDoS |
| **查询复杂度** | 摊还 O(1)，实际接近常数（SIMD 加速） | 平均 O(1)，最坏 O(n) |

**边界示例**:

```rust
use std::collections::HashMap;

// ✅ 预分配减少 rehash
let mut map = HashMap::with_capacity(1000);
for i in 0..1000 {
    map.insert(i, i * 2);  // 避免扩容 rehash
}

// ✅ 自定义 hasher（如性能敏感且输入可信时）
// use std::hash::BuildHasherDefault;
// use hashbrown::hash_map::DefaultHashBuilder; // 或其他非加密 hasher
// let map: HashMap<i32, i32, BuildHasherDefault<FxHasher>> = HashMap::default();

// ❌ 反例: 默认 HashMap 不适合密码学场景
// HashMap 默认使用 SipHash 1-3，抗 HashDoS 但非加密安全
// 切勿用 HashMap 的 hash 输出作为密码学哈希或 MAC
```

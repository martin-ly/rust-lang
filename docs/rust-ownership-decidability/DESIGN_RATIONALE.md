# 设计决策原理：为什么选择这样的设计

> **本文档目标**：解释形式化工作中每个关键设计决策背后的原理、权衡和理由。

---

## 一、整体设计哲学

### 1.1 设计目标

**首要目标：可理解性 > 完备性**

```text
选择：
  ✓ 覆盖安全子集的完整理论
  ✗ 覆盖 unsafe 的复杂理论

理由：
  - 安全子集是 Rust 的核心
  - 完备理论太复杂，难以理解
  - 可理解的形式化更有价值
```

**次要目标：教学性 > 前沿性**

```text
选择：
  ✓ 基于成熟的 Featherweight Rust
  ✗ 最新的研究进展

理由：
  - 成熟理论更容易验证
  - 教学需要稳定性
  - 前沿研究变化太快
```

### 1.2 设计原则

**原则1：层次清晰**

```text
第0层：数学基础（归纳、良基）
第1层：元模型（语法、语义域）
第2层：理论基础（Linearizability）
第3层：核心定理（终止性等）
第4层：综合定理（类型安全）

原理：每层只依赖下层，保证理论坚固
```

**原则2：概念正交**

```text
所有权：谁拥有资源（空间）
借用：如何临时访问（权限）
生命周期：访问有效期（时间）

原理：三个概念正交，组合产生涌现性质
```

**原则3：证明模块化**

```text
每个定理独立证明
引理可重用
admit 可局部填充

原理：降低复杂度，便于协作
```

## 二、语法设计的原理

### 2.1 表达式语法的选择

**设计**：

```coq
Inductive expr :=
  | EValue : value -> expr
  | EVar : var -> expr
  | EBorrow : lifetime -> mutability -> place -> expr
  | EDeref : expr -> expr
  | EBox : expr -> expr
  | ESeq : expr -> expr -> expr
  | ELet : mutability -> var -> ty -> expr -> expr -> expr
  | EAssign : place -> expr -> expr
  | ETuple : list expr -> expr
  | EIf : expr -> expr -> expr -> expr
  | ECall : fn_name -> list ty -> list expr -> expr
  | ...
```

**为什么这样设计？**

**选择1：区分 EValue 和 EVar**

```text
EValue：字面量（5, true, "hello"）
EVar：变量引用

原理：
- 求值规则不同
- 类型检查不同
- 明确区分常量和变量
```

**选择2：EBorrow 包含 lifetime 和 mutability**

```text
EBorrow : lifetime -> mutability -> place -> expr

原理：
- 生命周期和可变性是引用的核心属性
- 显式包含，便于类型检查
- 与 Rust 的 &ρ ω T 对应
```

**选择3：ELet 包含 mutability**

```text
ELet : mutability -> var -> ty -> expr -> expr -> expr
     ↑
  可变性标记

原理：
- let mut x = ... vs let x = ...
- 可变性是绑定的一部分
- 影响后续借用检查
```

**放弃的设计**：

```coq
(* 放弃：过于复杂的控制流 *)
| EWhile : expr -> expr -> expr
| EFor : var -> expr -> expr -> expr
| EMatch : expr -> list (pattern * expr) -> expr

理由：
- 循环和模式匹配增加复杂度
- 不影响核心理论（所有权、借用）
- 可以在扩展中添加
```

### 2.2 类型语法的选择

**设计**：

```coq
Inductive ty :=
  | TBase : base_ty -> ty
  | TRef : lifetime -> mutability -> ty -> ty
  | TBox : ty -> ty
  | TTuple : list ty -> ty
  | TStruct : type_name -> list ty -> ty
  | TEnum : type_name -> list ty -> ty
  | TFn : list ty -> ty -> ty
```

**关键决策**：

**决策1：TRef 包含 lifetime**

```text
TRef : lifetime -> mutability -> ty -> ty

替代设计：
  TRef : mutability -> ty -> ty  （不包含生命周期）

选择原因：
  ✓ 生命周期是引用类型的核心部分
  ✓ 与 Rust 的 &'a T 对应
  ✓ 便于生命周期检查

代价：
  - 类型更复杂
  - 需要处理生命周期参数
```

**决策2：Box 是独立的类型构造子**

```text
TBox : ty -> ty

替代设计：
  用 TRef 表示堆分配

选择原因：
  ✓ Box 有独特的所有权语义
  ✓ 与引用不同（Box 拥有，引用借用）
  ✓ 便于表达 unique ownership
```

**决策3：区分 TStruct 和 TEnum**

```text
TStruct : 乘积类型（product type）
TEnum : 和类型（sum type）

原理：
- 对应 Rust 的 struct 和 enum
- 不同构造方式
- 不同内存布局（虽然形式化不关注）
```

### 2.3 为什么要简化语法

**与真实 Rust 的对比**：

```rust
// 真实 Rust
impl<T> Option<T> {
  fn map<U, F: FnOnce(T) -> U>(self, f: F) -> Option<U> {
    match self {
      Some(x) => Some(f(x)),
      None => None,
    }
  }
}
```

```coq
(* 形式化（简化） *)
EFunction "map" [T; U] [TFn [T] U] (EMatch ...)
```

**简化原则**：

1. **保留核心语义**

   ```text
   保留：所有权转移、借用、生命周期
   简化：泛型、trait、impl
   ```

2. **去除语法糖**

   ```text
   保留：let 绑定
   去除：if let, while let, match 守卫
   ```

3. **有限但足够**

   ```text
   足够表达核心理论
   有限以保持可管理
   ```

**简化带来的好处**：

- 证明更简洁
- 错误更少
- 教学更清晰

**简化的代价**：

- 不能直接验证真实 Rust 程序
- 需要扩展才能覆盖更多语言特性

## 三、语义设计的原理

### 3.1 为什么使用操作语义

**选择**：大步操作语义（Big-Step）+ 小步操作语义（Small-Step）

**替代选择**：

| 语义风格 | 优点 | 缺点 | 选择 |
|---------|------|------|------|
| 操作语义 | 直观，易理解 | 对并发建模困难 | ✓ |
| 指称语义 | 数学优雅 | 抽象，难理解 | ✗ |
| 公理语义 | 适合验证 | 不直观 | ✗ |

**操作语义的原理**：

```text
操作语义 ≈ "抽象解释器"

直观：描述程序如何执行
教学：容易理解
证明：适合结构归纳
```

### 3.2 大步 vs 小步的选择

**设计：两者都定义，证明等价性**

**大步语义（eval）**：

```coq
eval : stack -> heap -> expr -> runtime_val -> heap -> Prop

(* eval s h e v h' 表示 e 在 (s,h) 下求值为 v，产生新堆 h' *)
```

**适用场景**：

- 证明保持性（看结果类型）
- 函数式风格
- 教学入门

**小步语义（step）**：

```coq
step : stack -> heap -> expr -> stack -> heap -> expr -> Prop

(* step s h e s' h' e' 表示 e 单步到 e' *)
```

**适用场景**：

- 证明进展（每一步都有下一步）
- 并发建模
- 命令式风格

**等价性定理的价值**：

```text
两种语义等价
  ↓
可以选择更方便的视角
  ↓
大步证明保持性
小步证明进展
组合得到类型安全
```

### 3.3 环境设计的原理

**三层环境**：

```coq
(* 1. 区域环境：生命周期参数 *)
Definition region_env := list (var * lifetime).

(* 2. 类型环境：变量类型 *)
Definition type_env := list (var * ty).

(* 3. 借用环境：活跃的借用 *)
Definition loan_env := list (var * borrow_info).
```

**为什么分三层？**

**原理1：关注点分离**

```text
Δ（区域环境）：只关心生命周期参数
Γ（类型环境）：只关心变量类型
Θ（借用环境）：只关心活跃的借用

每层独立演化，互不干扰
```

**原理2：与 Rust 实现对应**

```text
Δ ≈ rustc 的 lifetime parameter context
Γ ≈ rustc 的 type context
Θ ≈ rustc 的 borrow checker state
```

**原理3：证明便利**

```text
可以独立证明每层性质
层间交互通过明确的接口
```

## 四、定理设计的原理

### 4.1 为什么选择这五个核心定理

**五个定理**：

1. 终止性（Termination）
2. 保持性（Preservation）
3. 进展（Progress）
4. 类型安全（Type Safety）
5. 可判定性（Decidability）

**选择的原理**：

**定理1-2-3：经典的 Safety + Liveness**

```text
保持性（Safety）："坏的事情不会发生"
进展（Liveness）："好的事情最终会发生"

经典组合，来自 Lamport 的时序逻辑
几乎所有 PL 形式化都包含
```

**定理4：类型安全（综合）**

```text
类型安全 = 保持性 + 进展

不引入新内容
但提供高层视角
便于理解
```

**定理5：可判定性（实用）**

```text
可判定性：类型检查可以自动化

理论价值：证明 Rust 编译器可实现
实用价值：保证编译器会终止
```

### 4.2 定理的依赖关系设计

**设计：层次化依赖，避免循环**

```text
数学基础
    ↓
Linearizability
    ↓
终止性
    ↓
可判定性

数学基础
    ↓
语义定义 + 类型系统
    ↓
保持性 + 进展
    ↓
类型安全
```

**避免循环依赖的原理**：

```text
循环依赖：
  A 依赖 B，B 依赖 A
  问题：无法确定证明顺序

层次依赖：
  每层只依赖下层
  清晰，可管理
```

### 4.3 为什么不证明更多定理

**放弃的定理**：

1. **完备性（Completeness）**

   ```text
   所有安全的程序都通过类型检查

   放弃原因：
   - Rust 故意不完备（保守）
   - 证明复杂，价值有限
   ```

2. **组合性（Compositionality）**

   ```text
   组件的安全组合成系统的安全

   放弃原因：
   - 需要模块系统形式化
   - 超出当前范围
   ```

3. **优化正确性**

   ```text
   编译器优化保持语义

   放弃原因：
   - 属于编译器验证，不是语言语义
   - 极度复杂
   ```

**设计的艺术**：

```text
足够覆盖核心理论
足够简洁可完成
足够实用有价值
```

## 五、证明设计的原理

### 5.1 为什么使用 Coq

**替代选择**：

| 工具 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| Coq | 生态成熟，教学好 | 语法繁琐 | ✓ |
| Isabelle/HOL | 自动化强 | 学习曲线陡峭 | ✗ |
| Agda | 现代，依赖类型 | 社区小 | ✗ |
| Lean 4 | 现代，性能好 | 相对新 | ✗ |

**选择 Coq 的原理**：

```text
1. 教学资源多（Software Foundations）
2. PL 形式化标准工具
3. 提取可执行代码
4. 社区支持好
```

### 5.2 证明风格的原理

**风格：详细但可读**

**示例对比**：

```coq
(* 风格1：极简，难理解 *)
Theorem t1: forall x, P x.
Proof. induction x; auto. Qed.

(* 风格2：详细，可读 *)
Theorem t1: forall x, P x.
Proof.
  (* 对 x 进行结构归纳 *)
  induction x.

  - (* 基本情况 *)
    simpl.
    constructor.

  - (* 归纳步骤 *)
    apply IH.
    auto.
Qed.
```

**选择风格2的原理**：

```
目标：教学 + 研究
需要：可读性 > 简洁性
代价：代码更长
收益：更容易理解和维护
```

### 5.3 admit 策略的原理

**设计：框架完整，细节用 admit**

**原理**：

```text
1. 先建立完整框架
2. 证明核心路径
3. 技术引理用 admit

理由：
- 框架完整性优先
- admit 的位置明确
- 可以并行填充
```

**admit 的分类**：

| 类型 | 数量 | 优先级 | 填充策略 |
|------|------|--------|---------|
| 技术性引理 | ~40 | 低 | 标准归纳 |
| 辅助函数性质 | ~15 | 中 | 函数展开 |
| 复杂 case 分析 | ~10 | 高 | 仔细分析 |

## 六、文档设计的原理

### 6.1 自然语言文档的设计

**设计：多层次文档体系**

```text
OVERVIEW：鸟瞰，建立整体认知
THEOREM_INTUITIONS：理解每个定理
PROOF_STRATEGIES：学习证明技巧
CONCEPT_MAP：建立概念联系
READING_GUIDE：个性化路径
```

**原理：认知负荷分层**

```text
不同读者：
  Rust 开发者 → 需要直观理解
  研究者     → 需要技术细节
  学生       → 需要学习路径

分层文档：
  每层针对特定需求
  可以按需阅读
  降低认知负荷
```

### 6.2 为什么写这么多自然语言

**数量：10个文档，~80,000字**

**原理**：

```text
形式化代码：机器可验证，人难理解
自然语言：人易理解

目标：两者都要
  - 机器验证正确性
  - 人类理解含义

投入：写自然语言时间 ≈ 写代码时间
收益：形式化工作的价值翻倍
```

### 6.3 文档与代码的对应

**设计：一一对应，双向导航**

```text
OVERVIEW 第2章 "所有权"
    ↓ 对应
Syntax/Types.v Definition owns

THEOREM_INTUITIONS 第1章 "终止性"
    ↓ 对应
Metatheory/Termination.v Theorem borrow_checking_termination
```

**原理：降低上下文切换成本**

```text
读者：
  在自然语言中理解概念
  查看形式化定义
  回到自然语言理解

需要：明确的对应关系
```

## 七、权衡与取舍

### 7.1 关键权衡

**权衡1：简洁 vs 完备**

```text
选择：简洁的安全子集
放弃：unsafe、复杂特性

理由：
  简洁 → 可完成、可理解
  完备 → 复杂、难以管理
```

**权衡2：形式化深度 vs 可读性**

```text
选择：中等深度 + 详细注释
放弃：极简或超详细

理由：
  中等深度 → 足够严谨
  详细注释 → 足够可读
```

**权衡3：理论创新 vs 教学价值**

```text
选择：整合现有 + 清晰解释
放弃：全新理论

理由：
  整合 → 可靠性高
  清晰解释 → 教学价值高
```

### 7.2 设计决策的回顾

**好的决策**：

- ✓ 分层设计
- ✓ 操作语义
- ✓ 详细自然语言文档
- ✓ 模块化证明

**可能的改进**：

- 可以更早添加更多示例
- 可以更早完成部分 admit
- 可以考虑使用更现代的 Coq 特性

## 八、设计原理的总结

### 8.1 核心设计原理

**原理1：层次化**

```text
数学 → 元模型 → 理论 → 定理 → 应用

每层清晰，依赖明确
```

**原理2：正交性**

```text
所有权、借用、生命周期正交

组合产生涌现性质
```

**原理3：双重性**

```text
形式化代码 + 自然语言
机器可验证 + 人可理解
```

**原理4：实用性**

```text
教学价值 > 前沿性
可理解性 > 完备性
```

### 8.2 设计决策的影响

**对读者的影响**：

- 容易理解核心理论
- 可以根据背景选择阅读路径
- 既有直观又有严格

**对形式化的影响**：

- 证明可管理
- 可扩展性好
- 可维护性强

**对领域的影响**：

- 可作为教学材料
- 可作为研究基础
- 可向工业界展示

---

**结语**：
> 每个设计决策都是权衡的结果。本工作的设计优先考虑可理解性和教学性，为 Rust 所有权系统的理解提供一个清晰的窗口。

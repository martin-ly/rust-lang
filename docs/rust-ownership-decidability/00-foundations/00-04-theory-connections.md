## 2. 线性逻辑 ↔ 分离逻辑

### 2.1 语法映射

| 线性逻辑 | 分离逻辑 | 含义 | Rust概念 |
|---------|---------|------|---------|
| `A ⊗ B` | `P * Q` | 分离合取 | 同时拥有的资源 |
| `A ⊸ B` | `P -* Q` | 分离蕴含 | 借用返回 |
| `!A` | 持久断言 □P | 可复制的 | Copy类型 |
| `?A` | 可能断言 ◇P | 可借用的 | 引用 |
| `1` | `emp` | 空资源 | () |
| `0` | `False` | 不可能 | ! (never) |

### 2.2 语义对应

```
线性逻辑的相空间语义 (Phase Space):
- 公式解释为相空间中的子集
- ⊗ 解释为乘法

分离逻辑的堆模型:
- 断言解释为堆集合
- * 解释为堆的不交并

对应:
相空间 ⊗  ↔  堆的并集
线性蕴涵 ⊸  ↔  magic wand -*
```

### 2.3 证明关系

```
定理: 线性逻辑的 MLL (Multiplicative Linear Logic)
      与分离逻辑的核心片段同构。

证明思路:
1. 从线性逻辑证明到分离逻辑:
   - ⊗-规则  映射到  *-规则
   - ⊸-规则  映射到  -*规则

2. 从分离逻辑到线性逻辑:
   - points-to ℓ ↦ v 编码为线性原子
   - * 直接对应 ⊗

3. 保持性:
   - 证明可转换
   - 归约保持
```

## 3. 仿射逻辑 ↔ Rust所有权

### 3.1 语法对应

```
仿射逻辑规则:
Γ ⊢ B
───────── (Weakening)
Γ, A ⊢ B

Rust实现:
let _unused = String::from("I can be ignored");
// 允许不使用变量

仿射逻辑规则:
Γ, A, A ⊢ B
─────────── (Contraction 被拒绝)
Γ, A ⊢ B

Rust实现:
let x = String::from("hello");
let y = x;
// let z = x;  // 错误！不能复制非Copy类型
```

### 3.2 可判定性论证

```
Kopylov (2001) 定理:
完整命题仿射逻辑是可判定的。

论证结构:
1. 弱化规则允许"忽略"公式
2. 这限制了证明搜索空间
3. 使用向量游戏语义，决策过程终止

Rust借用检查的可判定性:
- 基于NLL的数据流分析
- 约束求解在有限域上
- 借用检查等价于求解约束系统

定理: Rust借用检查（不包括泛型约束）是可判定的。
```

### 3.3 实践映射

| 仿射逻辑 | Rust概念 | 示例 |
|---------|---------|------|
| 弱化 (Weakening) | 允许未使用变量 | `let _x = 5;` |
| 无收缩 | Move语义 | `let y = x;` 后x不可用 |
| 仿射函数 | 消耗所有权的函数 | `fn f(s: String)` |
| !A (指数) | Copy类型 | `i32`, `bool` |

## 4. 分离逻辑 ↔ RustBelt/Iris

### 4.1 Iris 的分离逻辑基础

```
Iris 扩展了传统分离逻辑:

传统分离逻辑:
- 断言: P, Q ::= emp | ℓ ↦ v | P * Q | P -* Q | ...

Iris 扩展:
- 高阶断言: 断言可以量化其他断言
- 模态算子: ▷ (Later), □ (Persistence), ◇ (Update)
- 不变量: P^N (命名空间N中的不变量)
- 资源代数: 自定义资源组合规则

这些扩展使得 Iris 能够处理:
- 高阶函数
- 并发程序
- 复杂的不变量
```

### 4.2 所有权谓词的编码

```
RustBelt 中的类型解释:

[[own_n τ]].own(t, [ℓ]) ≡
  ∃v̄. ℓ ↦ v̄ * ▷[[τ]].own(t, v̄) * ▷Dealloc(ℓ, n, |τ|)

组件分解:
1. ℓ ↦ v̄: 分离逻辑的 points-to
2. ▷[[τ]].own(t, v̄): Later 模态，递归guard
3. Dealloc(ℓ, ...): 资源代数，释放权限

这展示了分离逻辑、模态逻辑、资源代数的融合。
```

### 4.3 生命周期逻辑的分离逻辑解释

```
生命周期作为区域:

&^α P 在 Iris 中表示:
- α: 生命周期 (区域变量)
- P: 借用的资源
- &^α: 在区域α上的借用

分离逻辑表示:
&^α P ≡ ∃t. [α]^t * (P 在 t 期间有效)

其中:
- [α]^t: α在时刻t存活的token
- 借用到期后，P返回原所有者
```

## 5. 类型理论 ↔ 程序验证

### 5.1 Curry-Howard 扩展

```
经典 Curry-Howard:
类型  ↔  逻辑命题
程序  ↔  证明

扩展到程序验证:
类型 + 规范  ↔  增强逻辑
程序         ↔  构造性证明
验证条件     ↔  证明义务

Rust的扩展:
所有权类型  ↔  线性/仿射逻辑
借用        ↔  模态逻辑
生命周期    ↔  时序逻辑
```

### 5.2 类型即规范

```rust
// 类型本身携带规范信息

// 所有权类型: T ⊸ U
fn transfer_ownership(s: String) -> usize {
    s.len()  // 消耗s，返回usize
}

// 借用类型: &T (共享), &mut T (独占)
fn borrow_shared(s: &String) -> usize {
    s.len()  // 只读访问
}

fn borrow_exclusive(s: &mut String) {
    s.push_str("!");  // 独占写访问
}

// 生命周期: 时序约束
fn with_lifetime<'a>(s: &'a str) -> &'a str {
    s  // 输出与输入同生命周期
}

// 这些类型签名就是规范！
```

## 6. 可判定性理论的统一视角

### 6.1 复杂度谱系

```
可判定性谱系:

可判定 ←────────────────────────────────────→ 不可判定

┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ 命题逻辑 │  │ HM推断   │  │ 仿射逻辑 │  │ 线性逻辑 │
│ coNP-C   │  │ O(n³)    │  │ TOWER    │  │ 不可判定 │
├──────────┤  ├──────────┤  ├──────────┤  ├──────────┤
│ Rust:    │  │ Rust:    │  │ Rust:    │  │ Rust:    │
│ 布尔类型 │  │ 类型推断 │  │ 核心借用 │  │ 泛型约束 │
│          │  │          │  │ 检查     │  │ 求解     │
└──────────┘  └──────────┘  └──────────┘  └──────────┘

Rust的设计: 将核心功能保持在可判定区域
```

### 6.2 近似与不可判定性的处理

```
Rust编译器处理不可判定性的策略:

1. 超时机制
   - 类型检查有最大时间限制
   - 超过则报错，提示代码过于复杂

2. 递归深度限制
   - 类型递归深度有限制
   - 宏展开深度有限制
   - 常量求值步数有限制

3. 保守近似
   - 借用检查可能过于保守
   - 接受所有安全程序是理想，但不可行
   - 实践中，逐步放宽 (NLL → Polonius)

4. 用户指导
   - 显式类型标注
   - 生命周期标注
   - where 子句
```

## 7. 范畴论语义

### 7.1 张量范畴

```
线性逻辑的范畴论语义:

*-自治范畴 (*-autonomous category):
- 对象: 类型
- 态射: 程序/证明
- ⊗: 张量积 (monoidal product)
- ⊸: 内部hom (线性函数)
- (_)^⊥: 对偶 (negation)

Rust近似于:
- 有向图范畴 (用于借用)
- 仿射范畴 (允许弱化)
```

### 7.2 仿射范畴

```
仿射范畴 (Affine Category):

结构:
- 每个对象A有弱化态射: A → 1 (终端对象)
- 允许"丢弃"对象
- 保持张量结构

对应Rust:
- Drop trait 实现弱化
- 允许未使用变量
- 但收缩仍然被禁止
```

### 7.3 有向图范畴

```
生命周期作为有向图:

对象: 程序点 (CFG节点)
态射: 控制流边

生命周期 'a:
- 从 'a 开始到 'a 结束的路径
- 形成子范畴

借用关系:
- 作为有向图中的约束
- 生命周期包含关系 = 子范畴关系
```

## 8. 综合论证：为什么 Rust 选择这些理论

### 8.1 设计决策分析

```
Rust的设计决策论证:

Q: 为什么选择仿射而非线性?
A:
- 线性过于严格，编程繁琐
- 允许未使用变量更符合直觉
- Drop trait 自动清理
- 保持可判定性 (Kopylov定理)

Q: 为什么使用分离逻辑而非纯类型?
A:
- 需要精确的内存模型
- 借用需要表达能力
- 分离逻辑已有成熟验证工具
- 模块化推理

Q: 为什么生命周期是显式的?
A:
- 类型推断无法完全自动推断
- 显式生命周期作为文档
- 与借用检查器交互
- 控制复杂度
```

### 8.2 理论选择的协同效应

```
仿射类型 + 分离逻辑 + 生命周期 = Rust所有权

协同效应:
1. 仿射类型确保资源管理
2. 分离逻辑处理借用
3. 生命周期处理时序
4. 三者结合，无需GC即可内存安全
5. 运行时零成本
```

## 9. 理论联系的实用意义

### 9.1 对编译器实现

```
理论指导实现:

- 借用检查器 = 求解约束系统 (来自NLL理论)
- 类型推断 = HM算法 (来自类型理论)
- 生命周期推断 = 数据流分析 (来自程序分析)
- 优化 = 基于类型的转换 (来自范畴论)
```

### 9.2 对形式化验证

```
理论指导验证:

- RustBelt 使用 Iris (分离逻辑)
- Creusot 使用预言变量 (时序逻辑)
- Prusti 使用 Viper (分离逻辑)
- Verus 使用资源代数 (代数逻辑)

这些工具的选择都与Rust的理论基础一致。
```

### 9.3 对教学与学习

```
理论指导学习:

- 理解线性/仿射类型 → 理解所有权
- 理解分离逻辑 → 理解借用
- 理解时序逻辑 → 理解生命周期
- 类型即规范 → 类型驱动开发
```

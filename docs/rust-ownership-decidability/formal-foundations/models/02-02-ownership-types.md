## 1. 引言：资源敏感的类型系统

### 1.1 资源管理问题

传统类型系统关注**值的形状**，资源敏感类型系统额外关注**值的使用次数**。

```text
传统类型系统的问题:

fun closeFile(f: FileHandle): Unit
fun readFile(f: FileHandle): String

问题: 以下代码类型正确但运行时错误!

let f = openFile("data.txt")
let content = readFile(f)
closeFile(f)
readFile(f)  // ← 运行时错误: 已关闭的文件
```

资源敏感类型系统的目标：

```text
在编译时确保:
1. 资源被正确初始化
2. 资源在使用时有效
3. 资源不重复释放
4. 资源最终被释放（可选）
```

### 1.2 类型系统的演变

```text
类型系统谱系:

┌─────────────────────────────────────────────────────────────────┐
│                        类型系统演变                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  简单类型系统 (λ演算)                                            │
│  ├── 只关心值的结构                                             │
│  └── 不关心资源使用                                             │
│           │                                                     │
│           ▼                                                     │
│  线性类型系统 (Girard, 1987)                                    │
│  ├── 每个资源必须恰好使用一次                                    │
│  └── 禁止弱化和收缩                                             │
│           │                                                     │
│           ▼                                                     │
│  仿射类型系统 (Kopylov, 2001)                                   │
│  ├── 允许弱化 (资源可不使用)                                     │
│  └── 禁止收缩 (资源最多使用一次)                                 │
│           │                                                     │
│           ▼                                                     │
│  Rust的所有权系统                                               │
│  ├── 默认为是仿射类型                                           │

```

## 2. 线性类型系统

### 2.1 线性逻辑基础

**线性逻辑**由Jean-Yves Girard于1987年提出，核心思想是：

> **命题消耗其假设**：使用一个假设后，它就消失了。

```text
结构规则比较:

经典/直觉逻辑:                    线性逻辑:
───────────────                  ───────────────
Γ, A, A ⊢ B                      (禁止)
──────────── 收缩                ────────────
Γ, A ⊢ B

Γ ⊢ B                            Γ ⊢ B
──────── 弱化                    ────────  (禁止)
Γ, A ⊢ B

Γ, A, B ⊢ C                      Γ, A, B ⊢ C
───────────── 交换               ─────────────  交换
Γ, B, A ⊢ C                      Γ, B, A ⊢ C
```

### 2.2 线性类型的语法

```text
线性类型语法:

τ ::= 1           单位类型 (空元组)
   |  τ₁ ⊗ τ₂     张量积 (线性对)
   |  τ₁ ⊸ τ₂     线性函数 (使用参数恰好一次)
   |  τ₁ & τ₂     内部选择 (with)
   |  τ₁ ⊕ τ₂     外部选择 (plus)
   |  !τ          Of-course (指数模态)
   |  ?τ          Why-not (指数模态)
   |  0           空类型
   |  ⊤           最大类型
   |  ⊥           最小类型

上下文:
Γ ::= · | Γ, x:τ   (线性上下文: 每个变量必须恰好使用一次)
```

### 2.3 线性类型的推导规则

**核心推导规则**:

```text
变量 (Axiom):
────────────
x:τ ⊢ x:τ

单位 (Unit):
────────────
· ⊢ ():1

张量积 (⊗):
Γ₁ ⊢ e₁:τ₁    Γ₂ ⊢ e₂:τ₂
────────────────────────────
Γ₁, Γ₂ ⊢ (e₁, e₂):τ₁ ⊗ τ₂

张量消去 (⊗-elim):
Γ₁ ⊢ e:τ₁ ⊗ τ₂    Γ₂, x:τ₁, y:τ₂ ⊢ e':τ'
────────────────────────────────────────────
Γ₁, Γ₂ ⊢ let (x,y) = e in e':τ'

线性函数 (⊸):
Γ, x:τ₁ ⊢ e:τ₂
────────────────
Γ ⊢ λx.e:τ₁ ⊸ τ₂

函数应用 (Application):
Γ₁ ⊢ e₁:τ₁ ⊸ τ₂    Γ₂ ⊢ e₂:τ₁
────────────────────────────────
Γ₁, Γ₂ ⊢ e₁ e₂:τ₂

内部选择 (&):
Γ ⊢ e₁:τ₁    Γ ⊢ e₂:τ₂
────────────────────────────
Γ ⊢ (e₁, e₂):τ₁ & τ₂

外部选择 (⊕):
Γ ⊢ e:τᵢ
────────────────
Γ ⊢ injᵢ e:τ₁ ⊕ τ₂
```

**指数模态** (允许非线性使用):

```text
Dereliction (!):
Γ, x:τ ⊢ e:τ'
──────────────────
Γ, x:!τ ⊢ e:τ'

Promotion (!):
!Γ ⊢ e:τ
────────────────
!Γ ⊢ !e:!τ

其中 !Γ = x₁:!τ₁, ..., xₙ:!τₙ
```

### 2.4 线性类型的语义

**Categorical语义** (幺半范畴):

```text
在*-autonomous category C中:

[[1]] = I                     (单位对象)
[[τ₁ ⊗ τ₂]] = [[τ₁]] ⊗ [[τ₂]] (张量积)
[[τ₁ ⊸ τ₂]] = [[τ₁]] ⊸ [[τ₂]] (线性函数对象)
[[!τ]] = ![[τ]]               (余自由余代数)

解释函数:
[[Γ ⊢ e:τ]] : [[Γ]] → [[τ]]
```

**游戏语义** (Game Semantics):

```text
类型τ的解释为游戏G_τ:
- 动作集合 (moves)
- 先后手关系
- 正当策略 (valid strategies)

线性函数 τ₁ ⊸ τ₂:
- 对手在 τ₂ 上扮演Opponent
- 对手在 τ₁ 上扮演Proponent
- 程序必须对 τ₁ 的每个输入恰好响应一次
```

## 3. 仿射类型系统

### 3.1 从线性到仿射

**仿射逻辑**是线性逻辑的扩展，允许**弱化规则**但不允许**收缩规则**。

```text
仿射逻辑的结构规则:

弱化 (Weakening) - 允许:
Γ ⊢ B
────────
Γ, A ⊢ B

收缩 (Contraction) - 禁止:
Γ, A, A ⊢ B
────────────
Γ, A ⊢ B    ✗

交换 (Exchange) - 允许:
Γ, A, B, Γ' ⊢ C
────────────────
Γ, B, A, Γ' ⊢ C
```

**Rust与仿射逻辑的对应**:

```rust
// 1. 弱化规则: 资源可以不使用
fn weakening_example() {
    let x = String::from("hello");
    // x 未被使用 - 允许!
    // Drop trait确保资源最终释放
}

// 2. 收缩规则: 资源不能多次使用 (禁止)
fn no_contraction() {
    let x = String::from("hello");
    let y = x;  // x 被移动
    // println!("{}", x);  // 错误! x 已被使用
}
```

### 3.2 仿射逻辑的判定

**Kopylov定理 (2001)**:

> 完整命题仿射逻辑 (Full Propositional Affine Logic) 是**可判定的**。

```text
复杂度结果:

┌─────────────────────────┬──────────────────┐
│ 逻辑系统                 │ 复杂度            │
├─────────────────────────┼──────────────────┤
│ 命题逻辑                 │ coNP-Complete    │
│ MALL (乘法加性线性逻辑)  │ PSPACE-Complete  │
│ 完整线性逻辑             │ 不可判定          │
│ 完整仿射逻辑             │ TOWER-Complete   │
└─────────────────────────┴──────────────────┘

TOWER-Complete 意味着复杂度是非初等递归的，但仍然是可判定的。
```

### 3.3 仿射类型的推导规则

```text
仿射类型系统的额外规则:

弱化 (Affine Weakening):
Γ ⊢ e:τ
──────────── (W)
Γ, x:τ' ⊢ e:τ

其他规则与线性类型相同，但上下文操作遵循仿射规则。

关键区别:
- 线性: 每个变量必须恰好使用一次
- 仿射: 每个变量最多使用一次 (0次或1次)
```

## 4. Rust的所有权系统

### 4.1 所有权作为仿射类型

**所有权的三条规则**:

```text
1. 每个值有且只有一个所有者
2. 当所有者离开作用域，值被丢弃
3. 所有权可以转移 (Move)

对应仿射逻辑:
- 规则1: 资源最多有一个引用 (无收缩)
- 规则2: 弱化规则 (资源可以不使用，但会被Drop)
- 规则3: 资源转移保持仿射性质
```

**形式化对应**:

```text
Rust所有权类型 ≈ 仿射类型 + Drop类型类

[[String]].affine(x) ≡ owns(x) * (owns(x) ⊸ Drop(x))
```

### 4.2 Copy类型作为线性类型

**Copy trait**标记的类型是**线性类型**：

```rust
// Copy类型: 线性行为 (可复制)
fn copy_types() {
    let n: i32 = 5;
    let m = n;  // 复制，n仍可用
    let p = n;  // 再次复制
    println!("{}", n);  // OK!
}

// 非Copy类型: 仿射行为
fn affine_types() {
    let s = String::from("hello");
    let t = s;  // 移动
    // println!("{}", s);  // 错误! s已被移动
}
```

**形式化**:

```text
Copy类型:    !τ  (指数的of-course)
非Copy类型:  τ   (仿射类型)

复制推导:
Γ ⊢ e:!τ
────────────────
Γ ⊢ copy(e):!τ    Γ ⊢ e:!τ

对比:
- !τ: 可以任意复制
- τ: 最多使用一次
```

### 4.3 Drop trait的形式化

**Drop trait的语义**:

```text
Drop trait定义了资源释放的行为:

trait Drop {
    fn drop(&mut self);
}

形式化语义:
[[Drop]].own(t, [ℓ]) ≡ ∃f. ℓ ↦ f * (∀v. call(f, v) {resource_freed(v)})
```

**Drop检查**:

```rust
// 隐式Drop
fn implicit_drop() {
    let x = String::from("hello");
} // x在这里自动drop

// 显式Drop (使用mem::drop)
fn explicit_drop() {
    let x = String::from("hello");
    drop(x);
    // x 不可用
}
```

```text
形式化Drop规则:

Γ ⊢ e:τ    Drop(τ)    x ∉ fv(Γ')
───────────────────────────────────
Γ ⊢ { drop(e); e' }:τ'
```

## 5. 形式化模型

### 5.1 类型判断的形式化

**扩展的类型判断**:

```text
Γ; Δ ⊢ e : τ

其中:
- Γ: 线性/仿射变量上下文
- Δ: 普通变量上下文
- e: 表达式
- τ: 类型
```

**所有权规则**:

```text
移动规则 (Move):
Γ, x:τ; Δ ⊢ e : τ'
────────────────────────────
Γ; Δ ⊢ let y = x in e : τ'

借用规则 (Borrow):
Γ; Δ ⊢ &x : &τ
────────────────
Γ; x:τ, Δ ⊢ &x : &τ

复制规则 (Copy):
Γ; x:!τ, Δ ⊢ e : τ'
────────────────────────────
Γ; x:!τ, Δ ⊢ let y = x in e : τ'
(注意: x保留在上下文中)
```

### 5.2 资源跟踪

**资源状态转换**:

```text
资源状态机:

        ┌─────────────┐
        │ Uninit      │
        └──────┬──────┘
               │ init
               ▼
        ┌─────────────┐
   ┌────│   Owned     │────┐
   │    └──────┬──────┘    │ move
   │borrow     │ drop      │
   ▼           ▼           ▼
┌──────┐   ┌──────┐   ┌──────┐
│Borrow│   │Dropped│  │Moved │
└──────┘   └──────┘   └──────┘
```

**路径跟踪**:

```text
路径表达式:
p ::= x | p.f | p[i] | *p

所有权状态:
owns(Γ, p) ≡ 路径p在上下文Γ中是未借用的

借用状态:
borrowed(Γ, p, kind) ≡ 路径p被借用，kind ∈ {Shared, Mut}
```

### 5.3 子类型关系

**所有权与子类型**:

```text
子类型关系:

τ <: τ'  表示 τ 可以安全地用作 τ'

规则:
1. 自反性: τ <: τ
2. 传递性: τ₁ <: τ₂  ∧  τ₂ <: τ₃  ⊢  τ₁ <: τ₃
3. 协变: Box<τ₁> <: Box<τ₂>  if τ₁ <: τ₂
4. 逆变: fn(τ₂) -> τ <: fn(τ₁) -> τ  if τ₁ <: τ₂

生命周期子类型:
&'a τ <: &'b τ  if 'a: 'b (生命周期包含)
```

## 6. 复杂性与可判定性

### 类型检查的复杂性

```text
Rust类型系统的复杂性:

┌─────────────────────────────┬─────────────────┐
│ 检查项目                     │ 复杂度          │
├─────────────────────────────┼─────────────────┤
│ 基础类型推断                 │ PTIME (O(n³))   │
│ 借用检查 (NLL)               │ PTIME           │
│ 生命周期约束求解             │ PTIME           │
│ 泛型约束求解 (Traits)        │ 图灵完备        │
│ Drop检查                     │ PTIME           │
│ const求值                    │ 图灵完备        │
└─────────────────────────────┴─────────────────┘
```

### 可判定性边界

```text
可判定 vs 不可判定:

可判定的:
├── 基础所有权检查
├── 借用检查 (不含高阶生命周期)
├── 简单生命周期推断
└── Drop标志分析

不可判定的 (一般情况):
├── 高阶类型推断
├── 复杂trait约束求解
├── 关联类型推导
└── 递归impl的终止性
```

## 8. 实践意义

### 内存安全保证

```text
Rust的所有权系统保证:

1. 无悬挂指针
   - 所有权转移确保引用有效
   - 生命周期检查确保引用不越界

2. 无双重释放
   - 所有权唯一性保证
   - Move语义防止重复drop

3. 无内存泄漏 (通过类型系统)
   - Drop trait确保资源释放
   - RAII模式自动管理

4. 无数据竞争
   - 借用规则: &mut XOR &
   - Send/Sync标记线程安全
```

### 性能优势

```text
零成本抽象:

传统GC:                    Rust所有权:
运行时间开销               编译时检查
内存开销 (追踪信息)         无运行时开销
暂停时间                   确定性的资源释放
```

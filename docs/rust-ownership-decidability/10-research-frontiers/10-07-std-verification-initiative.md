## 1. 项目概述

### 1.1 背景与动机

Rust 标准库是 Rust 生态系统的基石，被无数应用程序依赖。
虽然 Rust 的类型系统提供了内存安全保证，但标准库的某些部分使用了 `unsafe` 代码来实现高性能抽象。
这些 `unsafe` 代码块的正确性依赖于人工审查，存在潜在风险。

**为什么要验证标准库?**

1. **基础安全**: 标准库的安全直接影响所有 Rust 程序
2. **信任根基**: 为 Rust 的安全承诺提供形式化基础
3. **工具评估**: 评估和推动验证工具的发展
4. **文档价值**: 规格说明作为精确文档

### 1.2 项目目标

**主要目标**:

- 为标准库核心模块添加形式化规格说明
- 使用自动化验证工具证明关键属性
- 建立可持续的验证工作流程

**设计公理** (Design Axioms):

| 公理 | 说明 |
|------|------|
| **零运行时开销** | 验证插桩不能影响运行时行为或性能 |
| **自动化验证** | 需要自动化验证以适应标准库的快速迭代 |
| **合约即代码** | 合约语言尽可能接近 Rust 语法 |

## 2. 验证工具选择

### 2.1 工具对比

项目评估了多个验证工具，每个都有其优势和限制:

| 工具 | 机构 | 方法 | 优势 | 限制 | 适用场景 |
|------|------|------|------|------|----------|
| **Kani** | Amazon | 有界模型检测 | 支持 unsafe，全自动 | 循环需展开，单态化 | 底层 unsafe 代码 |
| **Creusot** | Inria | 演绎验证 (Why3) | 支持泛型，高度自动化 | 不支持 unsafe | Safe 抽象层 |
| **Verus** | CMU/VMware | SMT + 资源代数 | 支持并发，系统设计 | 需扩展 Rust 语法 | 并发原语 |

### 2.2 工具分工

```
标准库验证工具分工:
┌─────────────────────────────────────────────────────────────┐
│                     Rust 标准库                             │
├───────────────┬───────────────────┬─────────────────────────┤
│   Unsafe 核心  │   Safe 抽象层      │    并发原语             │
│   (raw ptr)   │   (Vec, String)   │    (Mutex, Arc)         │
├───────────────┼───────────────────┼─────────────────────────┤
│    Kani       │     Creusot       │      Verus              │
│  模型检测     │   演绎验证        │    系统验证             │
└───────────────┴───────────────────┴─────────────────────────┘
```

**互补策略**:

- 不同工具验证不同模块
- 交叉验证关键属性
- 综合各工具优势

## 3. 合约语言设计

### 3.1 设计原则

**目标**: 创建一个统一的合约规范语言，兼容多个验证工具。

**原则**:

1. **Rust 原生**: 语法尽可能接近 Rust
2. **可移植**: 可被多个验证工具理解
3. **零开销**: 编译时擦除，无运行时成本
4. **增量采用**: 可以逐步添加到现有代码

### 3.2 语法提案

基于 MCP (Major Change Proposal) 的讨论，提议的合约语法:

```rust
// 前置条件
#[requires(index < self.len())]
// 后置条件
#[ensures(result == old(self[index]))]
// 不变量
#[invariant(self.len() <= self.capacity())]
```

**更复杂的示例**:

```rust
impl<T> Vec<T> {
    /// # Safety
    /// `index` must be within bounds
    #[requires(index < self.len())]
    /// The returned reference is valid and points to the element
    #[ensures(*result == old(self[index]))]
    pub unsafe fn get_unchecked(&self, index: usize) -> &T {
        // ...
    }

    /// Push an element
    /// Length increases by 1
    #[ensures(self.len() == old(self.len()) + 1)]
    /// The pushed element is at the end
    #[ensures(self[self.len() - 1] == elem)]
    /// Previous elements unchanged
    #[ensures(
        forall(|i: usize| i < old(self.len())
               ==> self[i] == old(self[i]))
    )]
    pub fn push(&mut self, elem: T) {
        // ...
    }
}
```

## 4. 验证范围与优先级

### 4.1 核心模块

**第一阶段** (高优先级):

| 模块 | 内容 | 选择工具 | 复杂度 |
|------|------|----------|--------|
| `core::ptr` | 原始指针操作 | Kani | 高 |
| `core::slice` | 切片操作 | Kani/Creusot | 中 |
| `alloc::vec` | Vec 实现 | Creusot/Kani | 高 |
| `alloc::string` | String 实现 | Creusot | 中 |

**第二阶段** (中优先级):

| 模块 | 内容 | 选择工具 | 复杂度 |
|------|------|----------|--------|
| `core::sync` | 同步原语 | Verus | 高 |
| `std::collections` | 其他集合 | Creusot | 中 |
| `core::iter` | 迭代器 | Creusot | 中 |

### 4.2 验证属性

**安全属性** (必须验证):

```
内存安全:
├── 无 use-after-free
├── 无 double-free
├── 无缓冲区溢出
├── 无不安全指针解引用
└── 无不安全类型转换

线程安全 (并发代码):
├── 无数据竞争
├── 正确同步
└── Send/Sync 契约遵守
```

**功能属性** (尽可能验证):

```
功能正确性:
├── 操作前后条件
├── 不变量保持
├── 等价性规范
└── 边界条件处理
```

## 5. 技术挑战

### 5.1 Unsafe 代码验证

**挑战**:

- Unsafe 代码依赖人工审查的不变量
- 指针别名关系复杂
- 与 safe Rust 的边界需要精确建模

**解决方案**:

- 使用 Kani 进行有界模型检测
- 为 unsafe 函数编写详细的前置/后置条件
- 建立 unsafe 代码规范指南

### 5.2 泛型代码验证

**挑战**:

- 泛型函数需要在类型参数上通用验证
- Trait bounds 需要编码到规格说明中
- 单态化可能导致验证爆炸

**解决方案**:

- Creusot 支持在 trait 级别验证
- 使用抽象谓词编码 trait 语义
- 选择代表性实例进行验证

### 5.3 性能考虑

**挑战**:

- 验证可能非常耗时
- 标准库代码量大
- CI 集成需要平衡验证覆盖率和速度

**解决方案**:

- 分层验证策略 (关键路径优先)
- 增量验证 (只验证修改的代码)
- 并行化验证过程

```
标准库验证工作流程:

┌─────────────────────────────────────────────────────────────┐
│  1. 模块选择                                                │
│     • 基于使用频率和安全关键性                               │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  2. 规格编写                                                │
│     • 分析代码不变量                                         │

└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  3. 工具适配                                                │

└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  4. 验证执行                                                │
│     • 运行验证工具                                           │
```

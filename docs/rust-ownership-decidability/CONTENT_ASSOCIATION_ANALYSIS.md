## 四、关联缺失识别与修复

### 4.1 已识别的关联缺失

#### 缺失 1: 理论基础 ↔ 实践代码

**问题**: 线性逻辑、分离逻辑、区域类型等理论基础与实践 Rust 代码之间的映射不够明确。

**影响**: 学习者难以理解为什么 Rust 要这样设计。

**修复方案**:

```
创建: FOUNDATIONS_TO_PRACTICE_BRIDGE.md
内容:
1. 线性逻辑 → 所有权移动
2. 分离逻辑 → 借用规则
3. 区域类型 → 生命周期
4. 仿射类型 → Copy/Move trait
5. 每章包含: 理论概念 | Rust 代码 | 形式化规则
```

#### 缺失 2: 定理 ↔ 编译器实现

**问题**: 形式化定理与 rustc 实际实现之间的联系不够明确。

**影响**: 开发者难以理解形式化的实际意义。

**修复方案**:

```
创建: THEOREM_TO_COMPILER_BRIDGE.md
内容:
1. 终止性定理 ↔ rustc borrow_check
2. 类型安全定理 ↔ rustc type_check
3. 可判定性定理 ↔ rustc 编译流程
4. 每个定理对应 rustc 的哪个模块
```

#### 缺失 3: 模式 ↔ 理论约束

**问题**: 设计模式的选择缺乏明确的理论依据。

**影响**: 开发者不知道何时使用什么模式。

**修复方案**:

```
增强: 11-design-patterns/README.md
添加:
1. 每个模式的理论基础
2. 模式适用的所有权场景
3. 模式违反的理论约束
4. 形式化安全证明
```

#### 缺失 4: 验证工具 ↔ 理论性质

**问题**: 验证工具能验证什么性质不够明确。

**影响**: 用户不知道选择哪个工具。

**修复方案**:

```
增强: 03-verification-tools/README.md
添加:
1. Miri ↔ 运行时安全 (无未定义行为)
2. Kani ↔ 状态空间验证
3. Creusot ↔ 功能正确性 (Hoare逻辑)
4. Prusti ↔ 合约验证 (分离逻辑)
5. Verus ↔ 系统验证 (类型系统)
```

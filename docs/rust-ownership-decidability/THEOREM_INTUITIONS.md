> **本文档目标**：深入解释每个核心定理的含义、重要性和证明思路，用自然语言和类比让复杂的数学概念变得可理解。

## 一、终止性定理（Termination Theorem）

### 1.1 定理陈述

```coq
Theorem borrow_checking_termination :
  forall Γ,
    Linearizable Γ ->
    exists Γ' n,
      borrow_check_iter Γ n = Some Γ' /\
      is_fixed_point Γ'.
```

**自然语言翻译**：
> 如果类型环境 Γ 满足 Linearizability 条件，那么借用检查算法一定会在有限步（n 步）内终止，并产生一个不动点结果 Γ'。

### 1.2 直观理解

**类比：整理书架**

想象你在整理一个书架：

- 每本书（类型）可能引用其他书（其他类型）
- 你按照某种规则整理，直到没有书需要再移动

**问题**：这个过程会结束吗？还是可能无限循环？

**答案**：如果书的引用关系没有环（Linearizability），整理一定会在有限步内结束。

### 1.3 为什么这个定理重要？

**实际意义**：

1. **编译器可用性**：如果借用检查可能无限运行，Rust 编译器就无法用于实际开发
2. **用户体验**：开发者需要编译器在合理时间内给出结果
3. **算法基础**：所有借用检查优化都依赖终止性保证

**理论意义**：

- 这是 Rust 类型系统"可判定性"的基础
- 证明了借用检查不是图灵完备的（不会陷入无限循环）

### 1.4 证明策略详解

**核心思想**：构造一个"度量"（measure），每次借用检查都严格减小这个度量。

**步骤1：定义类型秩（Type Rank）**

```text
ty_rank(i32) = 0
ty_rank(&ρ T) = 1 + ty_rank(T)
ty_rank(Box T) = 1 + ty_rank(T)
```

**直观**：类型的"嵌套深度"

**步骤2：定义环境度量**

```text
te_measure(Γ) = Σ ty_rank(τ) for all (x,τ) in Γ
```

**步骤3：证明秩递减**

```text
Linearizable(Γ) =>
  如果 x: τ 在 Γ 中，且 y 被 τ 引用，
  那么 ty_rank(y的类型) < ty_rank(τ)
```

**步骤4：应用良基归纳**

因为自然数是良基的（没有无限递减链），所以过程必须终止。

### 1.5 常见疑问

**Q：所有 Rust 程序都满足 Linearizability 吗？**

- A：不是。但编译器会拒绝不满足的程序。Linearizability 是借用检查的前提条件。

**Q：终止性证明给出具体的时间复杂度了吗？**

- A：终止性只证明"有限"，不给出具体上界。实际复杂度分析见 `COMPLEXITY_ANALYSIS.md`。

## 二、保持性定理（Preservation Theorem）

### 2.1 定理陈述

```coq
Theorem preservation :
  forall Δ Γ Θ s h e τ s' h' v,
    has_type Δ Γ Θ e τ ->
    stack_well_typed s Γ ->
    eval s h e v h' ->
    exists Γ' Θ',
      has_type_value Δ Γ' Θ' v τ /\
      stack_well_typed s' Γ' /\
      heap_well_typed h' Θ'.
```

**自然语言翻译**：
> 如果表达式 e 在环境 Γ 下具有类型 τ，并且 e 求值为 v，那么 v 也具有类型 τ（可能在一个扩展后的环境 Γ' 中）。同时，执行后的栈和堆仍然保持良好类型。

### 2.2 直观理解

**类比：化学实验**

想象你做化学实验：

- 你把水（H₂O）和二氧化碳（CO₂）混合
- 如果操作正确，产物应该是碳酸（H₂CO₃）
- 保持性定理说："你得到的产物确实是你期望的类型"

**类比：函数计算**

```rust
let x: i32 = 5;
let y: i32 = x + 1;  // y 必须是 i32
```

保持性定理保证：`x + 1` 的结果确实是 `i32` 类型。

### 2.3 为什么这个定理重要？

**类型系统的核心保证**：

- 类型在运行时不会改变
- 变量的类型声明是可信的
- 函数返回值的类型是正确的

**没有这个定理会怎样？**：

```rust
let x: i32 = some_function();
// 如果保持性不成立，some_function 可能返回一个字符串！
```

### 2.4 证明策略详解

**核心方法**：对表达式 e 的结构进行归纳。

**基本情况**：e 是值

```text
e = EValue v

目标：证明 v 有类型 τ
方法：直接应用 value_has_type 的定义
```

**归纳步骤**：各种表达式构造

**Case: 变量**

```text
e = EVar x

已知：te_lookup Γ x = Some τ（变量在环境中有类型 τ）
求值：stack_lookup s x = Some v（栈中找到值 v）
目标：证明 v 有类型 τ

方法：
1. stack_well_typed 保证栈中值的类型与环境一致
2. 因此 v 确实有类型 τ
```

**Case: 借用**

```text
e = EBorrow ρ ω p

目标：证明创建的引用有类型 &ρ ω τ

方法：
1. place_has_type 保证位置 p 有类型 τ
2. ownership_safe 保证借用是合法的
3. 因此 &ρ ω p 有类型 &ρ ω τ
```

**Case: Let 绑定**

```text
e = ELet ω x τ₁ e₁ e₂

目标：证明 e₂ 的结果有类型 τ₂

方法：
1. 归纳假设：e₁ 的结果有类型 τ₁
2. 扩展环境 Γ' = Γ, x:τ₁
3. 在新环境下证明 e₂ 有类型 τ₂
```

### 2.5 技术难点

**难点1：环境扩展**

- 每次 Let 绑定都会扩展环境
- 需要证明扩展后的环境仍然保持良好类型

**难点2：堆更新**

- 赋值操作会修改堆
- 需要证明堆更新后仍然类型良好

**难点3：生命周期子类型**

- Rust 允许生命周期协变
- 需要证明子类型关系保持

## 三、进展定理（Progress Theorem）

### 3.1 定理陈述

```coq
Theorem progress :
  forall Δ Γ Θ s h e τ,
    has_type Δ Γ Θ e τ ->
    stack_well_typed s Γ ->
    heap_well_typed h Θ ->
    (is_exp_value e = true) \/
    (exists s' h' e', step s h e s' h' e') \/
    is_stuck e.
```

**自然语言翻译**：
> 如果表达式 e 有类型 τ，那么在良好类型的环境中，e 要么：
>
> 1. 已经是一个值，或者
> 2. 可以进一步求值，或者
> 3. 处于"卡住"状态（stuck）

**强进展定理**（更强的版本）：
> 良类型的表达式不会卡住。

### 3.2 直观理解

**类比：走迷宫**

想象你在走一个有标记的迷宫：

- **值**：你已经到达出口
- **可以求值**：你至少有一条路可以走
- **卡住**：你走进了一个没有出口的角落

**进展定理说**：只要你按照"良好类型"的规则走，你就不会走进死胡同。

**类比：计算过程**

```rust
let x = 5 + 3;  // 可以求值：5 + 3 = 8
let y = x * 2;  // 可以求值：8 * 2 = 16
y               // 已经是值：16
```

进展定理保证：每一步都有下一步（直到成为值）。

### 3.3 为什么这个定理重要？

**程序不会"神秘停止"**：

- 没有进展保证，程序可能突然无法继续
- 这是"定义良好"的程序的基本要求

**与保持性的配合**：

- 保持性：类型不会改变
- 进展：程序不会卡住
- 合起来：程序要么正常终止，要么可以继续安全执行

**实际意义**：

- 类型检查可以提前发现"卡住"的程序
- 运行时不会遇到未定义行为

### 3.4 证明策略详解

**核心方法**：对表达式的结构进行归纳。

**Case: 值**

```text
e = EValue v

目标：证明 e 是值
方法：直接应用定义，is_exp_value (EValue v) = true
```

**Case: 变量**

```text
e = EVar x

目标：证明变量可以求值

方法：
1. stack_well_typed 保证 x 在栈中有定义
2. 因此可以使用 S_Var 规则求值为栈中的值
```

**Case: 借用**

```text
e = EBorrow ρ ω p

目标：证明借用表达式可以求值

方法：
1. place_has_type 保证位置 p 是有效的
2. eval_place 可以将 p 求值为内存位置 ℓ
3. 因此借用可以求值为 RVLoc ℓ
```

**Case: Let**

```text
e = ELet ω x τ₁ e₁ e₂

目标：证明 Let 可以进展

子情况1：e₁ 是值
  → 使用 S_Let 规则，扩展环境，继续执行 e₂

子情况2：e₁ 可以求值
  → 在 e₁ 上进展，保持 e₂ 不变
```

**Case: 卡住的情况**

某些表达式可能是卡住的，例如：

- 解引用空指针
- 访问未定义的变量

但在**强进展定理**中，我们证明良类型的程序不会遇到这些情况。

### 3.5 卡住（Stuck）vs 错误（Error）

**区别**：

- **卡住**：没有定义如何继续求值（如除以零）
- **错误**：求值返回错误状态

在形式化中：

```coq
Definition is_stuck (e : expr) : Prop :=
  ~ is_exp_value e = true /\
  ~ (exists s' h' e', step s h e s' h' e').
```

卡住意味着：既不是值，又不能求值。

## 四、类型安全定理（Type Safety）

### 4.1 定理陈述

```coq
Theorem type_safety :
  forall Δ Γ Θ s h e τ,
    has_type Δ Γ Θ e τ ->
    stack_well_typed s Γ ->
    heap_well_typed h Θ ->
    forall s' h' v,
      eval s h e v h' ->
      exists Γ' Θ',
        has_type_value Δ Γ' Θ' v τ /\
        stack_well_typed s' Γ' /\
        heap_well_typed h' Θ'.
```

**自然语言翻译**：
> 良类型的程序求值后，结果仍然是良类型的。

### 4.2 直观理解

**类型安全 = 保持性 + 进展**

```text
保持性：类型在求值过程中保持不变
进展：程序不会卡住
-----------------------------------
类型安全：程序安全地执行到完成
```

**类比：保险箱**

- **保持性**：放入保险箱的东西，取出来时还是原来的东西
- **进展**：保险箱总是能打开（不会永远锁住）
- **类型安全**：你总能安全地取出你放入的东西

### 4.3 为什么这个定理是"圣杯"

类型安全定理是编程语言理论中的"圣杯"，因为它保证了：

1. **没有类型错误**：程序不会因为类型不匹配而崩溃
2. **内存安全**：没有 use-after-free，没有 dangling pointers
3. **可预测性**：程序的行为可以从类型推断

**形式化中的类型安全**：

```text
类型安全 ⟹ 没有 stuck 状态
类型安全 ⟹ 没有未定义行为
类型安全 ⟹ 内存安全
```

### 4.4 与 Rust 的关系

Rust 的类型系统就是围绕类型安全设计的：

```rust
// Rust 编译器拒绝这个程序
let x: &i32;
{
    let y = 5;
    x = &y;  // 错误！y 的生命周期不够长
}
// x 在这里无效
```

类型系统通过**生命周期检查**确保引用始终有效，这是类型安全的一部分。

## 五、可判定性定理（Decidability）

### 5.1 定理陈述

```coq
Theorem rust_type_system_decidable :
  forall Δ Γ Θ e τ,
    Linearizable Γ ->
    {has_type Δ Γ Θ e τ} + {~ has_type Δ Γ Θ e τ}.
```

**自然语言翻译**：
> 存在一个算法，可以在有限时间内判断任意表达式 e 是否具有类型 τ（假设环境满足 Linearizability）。

### 5.2 直观理解

**类比：考试评分**

- 可判定性：老师总能给出"对"或"错"的评分
- 不可判定：有些答案老师无法判断对错

**Rust 编译器的工作**：

```text
输入：Rust 程序
输出：
  - "编译通过"（程序类型正确）
  - 或错误信息（程序类型不正确）

可判定性保证：编译器总能给出答案（不会无限运行）
```

### 5.3 为什么这个定理重要？

**实际意义**：

1. **编译器可实现**：类型检查可以完全自动化
2. **开发者体验**：编译器在合理时间内给出结果
3. **IDE 支持**：实时类型检查成为可能

**理论意义**：

- Rust 类型系统不是图灵完备的
- 类型检查是一个"可计算"的问题

### 5.4 证明策略

**核心思想**：构造一个算法（函数）来进行类型检查。

**算法结构**：

```text
type_check(Δ, Γ, Θ, e, τ):
  match e with
  | EValue v -> check v has type τ
  | EVar x -> check Γ(x) = τ
  | EBorrow p ->
      check p has type τ'
      check τ = &ρ ω τ'
  | ESeq e1 e2 ->
      infer τ1 for e1
      check e2 has type τ
  | ...
```

**终止性**：

- 每个 case 都是有限的
- 递归调用在更小的表达式上
- Linearizability 保证借用检查终止

## 六、借用检查等价性定理

### 6.1 定理陈述

```coq
Theorem borrow_check_equivalent_to_ownership_safety :
  forall Γ e,
    borrow_check_expr Γ e = Success <->
    (forall Δ Θ, ownership_safe_program Δ Γ Θ e).
```

**自然语言翻译**：
> 表达式 e 通过借用检查，当且仅当 e 是所有权安全的（在所有可能的环境中都不会违反所有权规则）。

### 6.2 直观理解

**这个定理是 Rust 的核心！**

它建立了：

```text
编译时检查（借用检查器） ⟺ 运行时安全（所有权安全）
```

**类比：考试与能力**

- 借用检查通过 ⟺ 学生有能力
- 借用检查失败 ⟺ 学生能力不足

理想情况下，考试（借用检查）能准确评估能力（所有权安全）。

### 6.3 双向证明的解释

**方向1：借用检查通过 ⟹ 所有权安全**
（可靠性，Soundness）

> 如果编译器说"通过"，程序确实不会违反所有权规则。

**方向2：所有权安全 ⟹ 借用检查通过**
（完备性，Completeness）

> 如果程序不会违反所有权规则，编译器会说"通过"。

**实际 Rust**：

- 追求可靠性（宁可误报，不错报）
- 不完备（有些安全的程序被拒绝）
- 这是设计选择，不是理论限制

### 6.4 为什么这个定理重要？

**编译器可信度的理论基础**：

- 开发者信任编译器的判断
- 这个定理证明编译器的判断是正确的

**形式化验证的基础**：

- 如果程序通过类型检查
- 那么程序是所有权安全的
- 因此程序是内存安全的

## 七、内存安全定理

### 7.1 定理陈述

```coq
Theorem rust_type_system_guarantees_memory_safety :
  forall Δ Γ Θ e τ,
    has_type Δ Γ Θ e τ ->
    memory_safe e.
```

**自然语言翻译**：
> 如果程序通过 Rust 的类型检查，那么这个程序是内存安全的。

### 7.2 内存安全的定义

```coq
Definition memory_safe (e : expr) : Prop :=
  (* 1. 没有 use-after-free *)
  (forall s h ℓ,
    heap_lookup h ℓ = None ->
    ~ exists s' h' v, eval s h e v h' /\ accesses_loc h' ℓ)
  /\
  (* 2. 没有 double-free *)
  (forall s h,
    count_drop_calls e <= count_allocations e)
  /\
  (* 3. 没有 dangling pointers *)
  (forall s h v h',
    eval s h e v h' ->
    all_refs_valid h' v).
```

### 7.3 直观理解

**内存安全的三个支柱**：

**支柱1：没有 use-after-free**

```text
错误示例：
  let x = Box::new(5);
  let y = &*x;
  drop(x);      // x 被释放
  println!("{}", y);  // 错误！使用了已释放的内存
```

**Rust 如何防止**：

- 所有权系统：drop(x) 后，y 的引用失效
- 编译器拒绝这个程序

**支柱2：没有 double-free**

```text
错误示例（C++）：
  int* p = new int(5);
  delete p;
  delete p;  // 错误！重复释放
```

**Rust 如何防止**：

- 每个值只有一个所有者
- 所有者负责释放
- 移动语义确保不会重复释放

**支柱3：没有 dangling pointers**

```text
错误示例（C）：
  int* p;
  {
    int x = 5;
    p = &x;
  }  // x 在这里被销毁
  // p 现在是 dangling pointer
```

**Rust 如何防止**：

- 生命周期系统
- 编译器检查引用的有效期

### 7.4 定理的重要性

**这是 Rust 的"杀手锏"**：

> "Rust 保证内存安全，而不需要垃圾回收器。"

这个定理是这个声明的形式化证明。

**实际意义**：

- 开发者可以信任 Rust 编译器
- 系统编程不再需要手动管理内存的风险
- 并发编程更安全（没有数据竞争）

## 八、语义等价性定理

### 8.1 定理陈述

```coq
Theorem big_step_equiv_star_step :
  forall s h e v h',
    eval s h e v h' <-> star_step s h e h' (EValue v).
```

**自然语言翻译**：
> 大步求值（直接得到结果）和小步求值（一步一步执行）是等价的。

### 8.2 直观理解

**两种视角**：

**大步语义**：

```text
输入: (e, s, h)
输出: (v, h')

就像一个"黑盒子"：你放入 e，得到 v
```

**小步语义**：

```text
(e₀, s₀, h₀) → (e₁, s₁, h₁) → (e₂, s₂, h₂) → ... → (v, h')

就像"逐步执行"：你可以观察每一步的状态
```

**等价性**：
> 两种视角描述的是同一个计算过程。

### 8.3 为什么需要两种语义？

**大步语义**：

- 适合证明**结果**的性质
- 例如：保持性、类型安全

**小步语义**：

- 适合证明**过程**的性质
- 例如：并发、顺序、优化

**等价性定理**：

- 允许我们在证明中切换视角
- 选择更方便的语义进行证明

### 8.4 证明的挑战

**挑战1：大步 ⟹ 小步**

- 需要将一个大步分解为多个小步
- 构造小步的序列

**挑战2：小步 ⟹ 大步**

- 需要将多个小步合并为一个大步
- 证明最终结果是正确的

## 九、定理间的逻辑关系

### 9.1 依赖图

```text
终止性 ──────┐
             ├──→ 可判定性
Lineariz. ───┘

保持性 ──────┐
             ├──→ 类型安全 ──→ 内存安全
进展 ────────┘

借用检查等价性 ──→ 所有权安全 ──→ 内存安全
```

### 9.2 证明顺序

**推荐的阅读/证明顺序**：

1. **终止性**：证明借用检查会终止
2. **可判定性**：构造类型检查算法
3. **保持性**：证明类型在求值中保持
4. **进展**：证明程序不会卡住
5. **类型安全**：组合保持性和进展
6. **借用检查等价性**：连接编译时和运行时
7. **内存安全**：综合所有结果

---

## 十、总结

这七个核心定理构成了 Rust 所有权系统的理论基础：

| 定理 | 核心保证 | 实际意义 |
|------|---------|---------|
| 终止性 | 借用检查会结束 | 编译器不会卡住 |
| 保持性 | 类型不改变 | 变量类型可信 |
| 进展 | 程序不会卡住 | 程序能正常执行 |
| 类型安全 | 类型 + 进展 | 程序安全运行 |
| 可判定性 | 类型检查可自动 | 编译器可实现 |
| 借用检查等价性 | 编译时 = 运行时 | 编译器可信 |
| 内存安全 | 无内存错误 | Rust 的卖点 |

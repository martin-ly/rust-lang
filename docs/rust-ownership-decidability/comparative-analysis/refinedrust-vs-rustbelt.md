## 1. 概述

### 1.1 项目背景

**RustBelt** (POPL 2018):

- **时间**: 2015-2018
- **机构**: MPI-SWS, 研究员 Ralf Jung
- **目标**: 建立 Rust 内存安全的机器检查证明
- **成果**: 首个 Rust 基础性形式化，λRust 语言

**RefinedRust** (PLDI 2024):

- **时间**: 2021-2024
- **机构**: MPI-SWS, ETH Zurich, Radboud University
- **目标**: 验证真实 Rust 代码的功能正确性
- **成果**: 首个支持 safe/unsafe 的基础性验证工具

### 1.2 核心目标对比

| 维度 | RustBelt | RefinedRust |
|------|----------|-------------|
| **主要目标** | 证明内存安全 | 验证功能正确性 |
| **抽象级别** | 理想化语言 (λRust) | 真实 Rust (MIR) |
| **自动化** | 手动证明 | 半自动验证 |
| **unsafe 支持** | 理论支持 | 实际验证 |
| **证明范围** | 标准库核心 API | 用户代码 + 标准库 |

## 3. 形式化模型对比

### 3.1 编程语言抽象

**RustBelt: λRust**

```rust
// λRust 语法 (简化)
exp ::= x                     // 变量
      | λx.e                  // 函数
      | e1 e2                 // 应用
      | ref e                 // 引用创建
      | !e                    // 解引用
      | e1 := e2              // 赋值
      | newref e              // 内存分配
      | ...
```

**简化假设**:

- 无限整数 (无溢出)
- 简化类型系统
- 无方法调用语法
- 无模式匹配

**RefinedRust: Radium**

```rust
// Radium (基于 MIR)
// 支持:
// - 完整 Rust 语法
// - 整数溢出检查
// - 精确内存布局
// - Place 表达式

// 示例: Vec::get_unchecked_mut
fn get_unchecked_mut<T>(
    self: &mut Vec<T>,
    index: usize
) -> &mut T {
    // 实际 MIR 操作
    let ptr = self.ptr.as_ptr();
    let elem_ptr = ptr.add(index);
    &mut *elem_ptr
}
```

**精确建模**:

- 有限整数 (有溢出检查)
- 结构体/枚举布局
- 原始指针操作
- Place 访问 (x.f, x[i] 等)

### 3.2 内存模型

**RustBelt 内存模型**:

```
简化模型:
- 内存是 地址 → 值 的映射
- 值是抽象数学对象
- 不考虑对齐、布局
- 不考虑整数溢出
```

**RefinedRust 内存模型 (Caesium)**:

```
详细模型:
- 字节级内存表示
- 结构体字段偏移计算
- 对齐检查
- 整数溢出语义
- 类型有效性不变量
```

**关键区别示例**:

```rust
// 整数溢出
let x: i32 = 2_000_000_000;
let y = x + x;  // 溢出!

// RustBelt: 无溢出 (无限整数)
// RefinedRust: 检测到溢出，panic
```

### 3.3 类型系统

**RustBelt 类型**:

```rust
// λRust 类型
τ ::= Int                  // 无限整数
    | Bool                 // 布尔值
    | ref τ                // 引用
    | box τ                // 拥有的指针
    | τ → τ                // 函数
```

**RefinedRust 类型 (精细化)**:

```rust
// 精细化类型
τ ::= int{i32}             // 32位整数，可能溢出
    | #v @ place_τ         // 值 v 在位置类型
    | &ρ τ                 // 共享引用，生命周期 ρ
    | &mut ρ (x, γ) @ τ    // 可变引用 + borrow name
    | blocked(ρ, τ)        // 阻塞类型
    | box τ                // 拥有的指针
```

**创新: Borrow Names**:

```rust
// 可变引用的数学值是对 (当前值, borrow name)
&mut T  refined by  (v, γ)

// borrow name γ 用于通信引用的最终值
// 允许 reasoning about 可变引用的副作用
```

## 4. 验证方法对比

### 4.1 证明自动化

**RustBelt: 完全手动**

```coq
(* RustBelt 证明示例: Vec::push 安全 *)
Lemma vec_push_safe :
  ∀ (v : val) (xs : list val) (x : val),
  {{{ is_vec v xs }}}
    vec_push v x
  {{{ RET (); is_vec v (xs ++ [x]) }}}.
Proof.
  (* 手动分离逻辑推理 *)
  iIntros (Φ) "Hvec HΦ".
  unfold vec_push.
  wp_lam.
  (* ... 数十行证明脚本 ... *)
Qed.
```

**RefinedRust: 半自动 (Lithium)**

```coq
(* RefinedRust 自动生成证明 *)
(* 用户注解 *)
#[rr::params("xs: list T", "x: T")]
#[rr::args("Vec(xs)", "#x")]
#[rr::ensures("result == Vec(xs ++ [x])")]
fn push(&mut self, value: T) { ... }

(* Lithium 自动类型检查 *)
(* 生成约 3000 步自动推理 *)
(* 约 95% 的纯 Coq 条件自动解决 *)
```

**自动化对比**:

| 方面 | RustBelt | RefinedRust |
|------|----------|-------------|
| 证明编写 | 完全手动 | 注解驱动 |
| 证明长度 | 数百行 | 数十行注解 |
| 证明时间 | 数周 | 数分钟 (机器) |
| 专业知识 | 需要 Iris 专家 | 需要形式化基础 |

### 4.2 用户工作流

**RustBelt 工作流**:

```
1. 将 Rust 代码手动转换为 λRust
   (需要理解 λRust 语法)

2. 在 Coq 中编写分离逻辑规范
   (使用 Iris 战术)

3. 手动证明所有引理
   (通常数百行证明)

4. 提取安全性保证
```

**RefinedRust 工作流**:

```
1. 在 Rust 代码中添加注解
   #[rr::requires(...)]
   #[rr::ensures(...)]

2. 运行 RefinedRust 前端
   cargo refinedrust verify

3. Lithium 自动类型检查
   (~95% 自动化)

4. 手动解决剩余条件
   (可选，通常 5-10 个)

5. 获得 Coq 证明
```

## 5. 能力对比

### 5.1 功能矩阵

| 能力 | RustBelt | RefinedRust |
|------|----------|-------------|
| **内存安全证明** | ✅ | ✅ |
| **功能正确性** | ⚠️ 理论 | ✅ 实际 |
| **Safe Rust** | ✅ | ✅ |
| **Unsafe Rust** | ✅ 理论 | ✅ 实际验证 |
| **并发** | ✅ | ❌ |
| **真实 Rust 语法** | ❌ | ✅ |
| **整数溢出检查** | ❌ | ✅ |
| **精确内存布局** | ❌ | ✅ |
| **自动化验证** | ❌ | ✅ (半自动) |
| **证明复用** | 困难 | 模块化 |

### 5.2 限制对比

**RustBelt 限制**:

1. **抽象差距**: λRust 与真实 Rust 有显著差距
2. **手动转换**: 需要人工将 Rust 转为 λRust
3. **时间成本**: 每个引理需要数周证明
4. **专家依赖**: 需要深入的 Iris/分离逻辑知识

**RefinedRust 限制**:

1. **语言覆盖**: 暂不支持并发、递归类型、traits
2. **自动化**: 仍需一些手动证明
3. **性能**: 验证耗时 (Vec API 约 6 分钟)
4. **成熟度**: 工具链不如 RustBelt 成熟

## 6. 设计决策分析

### 6.1 为什么 RefinedRust 支持 Places

**问题**: Rust 的 "places" (也称为 lvalues) 是什么？

```rust
// Places 出现在:
let x = &mut obj.field;  // obj.field 是 place
arr[index] = value;      // arr[index] 是 place
*ptr = new_value;        // *ptr 是 place
```

**RustBelt 的处理**:

- 不处理一般 places
- 要求手动替换为简单情况
- 例如：将 `&mut obj.field` 转为 `&mut tmp`，其中 `tmp = obj.field`

**RefinedRust 的处理**:

- 原生支持 places
- 引入 place types 系统
- 直接反映 Rust 源代码结构

**影响**:

- RefinedRust 可以验证更真实的代码
- 但类型系统更复杂
- 需要处理 field offsets、数组索引等

### 6.2 为什么 RustBelt 使用 λRust

**设计决策原因**:

1. **聚焦核心问题**: 证明内存安全，不需要所有 Rust 特性
2. **简化证明**: 理想化语言更容易推理
3. **理论贡献**: 专注于所有权系统的核心逻辑
4. **时间限制**: 2018 年前完成

**权衡**:

- ✅ 成功证明核心抽象的安全性
- ❌ 难以扩展到真实代码验证

### 6.3 自动化 vs 基础性

**RustBelt 选择**: 基础性 + 手动

- 机器检查证明，无自动化
- 最高保证级别
- 适合核心库证明

**RefinedRust 选择**: 基础性 + 半自动

- 仍然机器检查
- 用户添加注解，机器验证
- 适合实际代码验证

**非基础性工具** (对比):

- **Kani**: 全自动，无机器检查证明
- **Creusot**: 半自动，Why3 验证

## 8. 选择指南

**选择 RustBelt 如果您**:

- 需要验证核心语言抽象的安全性
- 有足够时间进行手动证明
- 有 Iris/分离逻辑专家
- 目标是发表理论研究

**选择 RefinedRust 如果您**:

- 需要验证真实 Rust 代码的功能正确性
- 涉及 unsafe 代码指针操作
- 需要基础性证明
- 可接受半自动化流程

**选择其他工具 如果您**:

- 需要快速验证 (Kani)
- 纯 safe Rust (Creusot/Prusti)
- 不需要基础性证明

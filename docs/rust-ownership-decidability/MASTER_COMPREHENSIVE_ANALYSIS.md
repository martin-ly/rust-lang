## 第二部分: 核心概念层次模型

### 2.1 五层抽象模型

#### Layer 0: 物理内存层

```text
概念: 地址 → 值 → 字节
示例: 0x7fff_1234 → 0x0000_0005 → [0x05, 0x00, 0x00, 0x00]
```

#### Layer 1: 逻辑内存层

```text
概念: 变量 → 绑定 → 作用域
示例: let x = 5;
     x 绑定到 值5
     作用域: 从声明到块结束
```

#### Layer 2: 所有权层

```text
概念: 所有权 → 转移 → 释放
关键问题: 谁拥有这个数据？
答案: 在任何时刻，只有一个所有者
```

#### Layer 3: 借用层

```text
概念: 引用 → 权限 → 限制
不可变引用: &T (只读)
可变引用: &mut T (读写)
```

#### Layer 4: 类型系统层

```text
概念: 泛型 → Trait → 生命周期
连接: 类型参数可以带有生命周期约束
```

### 2.2 层次间交互规则

```text
规则 1: 下层决定上层 (物理限制逻辑)
  示例: 物理地址不能同时被两个可变引用访问

规则 2: 上层约束下层 (逻辑规范物理)
  示例: 生命周期约束决定何时释放内存

规则 3: 同层不跨越 (层内隔离)
  示例: 借用检查不直接操作物理地址

规则 4: 上下双向映射 (可追踪)
  示例: 类型错误可以映射到具体的内存安全问题
```

## 第三部分: 所有权系统完整论证

### 3.1 所有权的三要素

```text
┌─────────────────────────────────────────┐
│          所有权 (Ownership)              │
├─────────────────────────────────────────┤
│                                         │
│  1. 唯一性 (Uniqueness)                  │
│     ├─ 定义: 每个值有且仅有一个所有者      │
│     ├─ 目的: 明确释放责任                 │
│     └─ 反例: C++ 共享指针 (引用计数)       │
│                                         │
│  2. 可转移性 (Transferability)           │
│     ├─ 定义: 所有权可以转移给新的绑定      │
│     ├─ 语法: let y = x; (Move)           │
│     └─ 例外: Copy trait (隐式复制)        │
│                                         │
│  3. 作用域绑定 (Scope-binding)            │
│     ├─ 定义: 所有者在作用域结束时释放资源  │
│     ├─ 机制: Drop trait 自动调用          │
│     └─ 控制: 可以手动提前释放 (drop())     │
│                                         │
└─────────────────────────────────────────┘
```

### 3.2 所有权转移的完整示例

```rust
// 场景 1: 基本转移
let s1 = String::from("hello");  // s1 拥有字符串
let s2 = s1;                      // 所有权转移到 s2
// println!("{}", s1);            // 错误! s1 不再有效

// 场景 2: 函数参数转移
fn take_ownership(s: String) {    // s 获得所有权
    println!("{}", s);
} // s 在这里被释放

let s = String::from("hello");
take_ownership(s);                // s 的所有权转移到函数
// println!("{}", s);            // 错误! s 已被移动

// 场景 3: 返回值转移
fn gives_ownership() -> String {
    String::from("yours")
}

let s = gives_ownership();        // 获得返回值的所有权
```

### 3.3 Copy vs Move 的深层逻辑

```text
┌────────────────────────────────────────┐
│        Copy trait 的语义               │
├────────────────────────────────────────┤
│                                        │
│  条件: 类型的值可以按位复制              │
│       且没有资源需要释放                │
│                                        │
│  示例类型:                              │
│    - 所有整数类型 (i32, u64, etc.)      │
│    - 浮点数 (f32, f64)                  │
│    - 布尔值 (bool)                      │
```

## 第四部分: 借用系统多层次分析

### 4.1 借用的本质

```text
借用不是所有权，而是临时访问权限。

类比:
  所有权 = 拥有房子 (可以装修、出售、拆除)
  不可变借用 = 参观许可 (只能看，不能改)
  可变借用 = 租赁 (可以住，可以装修，但不能出售)
```

### 4.2 借用规则的逻辑推导

```text
目标: 防止数据竞争 (Data Race)

数据竞争的条件:
  1. 两个或更多指针同时访问同一数据
  2. 至少一个指针用于写入
  3. 没有同步机制

Rust 的解决方案:
  规则: 要么多个不可变借用，要么一个可变借用

逻辑推导:
  情况 1: 多个 &T
    - 都是读取，没有写入
    - 不会导致数据竞争 ✓

  情况 2: 一个 &mut T
    - 独占写入，没有其他访问
    - 不会导致数据竞争 ✓

  情况 3: &T 和 &mut T 同时存在
    - 有读取也有写入
    - 可能导致数据竞争 ✗

  情况 4: 多个 &mut T
    - 多个写入
    - 可能导致数据竞争 ✗
```

### 4.3 借用检查器的决策树

```text
                借用请求 (&x 或 &mut x)
                        │
            ┌───────────┴───────────┐
            │                       │
        不可变借用 (&x)          可变借用 (&mut x)
            │                       │
    ┌───────┴───────┐       ┌───────┴───────┐
    │               │       │               │
检查是否存在        │       │         检查是否存在
可变借用？          │       │         任何借用？
    │               │       │               │
   是 → 错误        │       │               是 → 错误
    │               │       │               │
   否              │       │               否
    │               │       │               │
  允许             │       │             允许
  多个读者         │       │           独占写入
```

### 4.4 借用与生命周期的关系

```text
借用不是永久的，它有一个有效期 (生命周期)。

生命周期的作用:
  1. 限制借用的有效范围
  2. 确保引用不会 dangling
  3. 允许编译时检查而非运行时检查

示例:
  let x = 5;
  let y = &x;  // y 的生命周期不能超过 x
  // y 在这里仍然有效
  // x 在这里仍然有效
  // 当 x 被释放时，y 自动失效
```

## 第五部分: 生命周期逻辑框架

### 5.1 生命周期的本质

```text
生命周期不是类型，而是对值有效范围的约束。

类比:
  变量 = 人
  生命周期 = 人的寿命
  引用 = 对人的引用 (比如电话号码)

  如果人死了 (变量释放)，电话号码就无效了 (dangling 引用)
  生命周期确保引用不会比被引用者活得更长
```

### 5.2 生命周期的三种形式

```text
┌─────────────────────────────────────────┐
│           生命周期的形式                │
├─────────────────────────────────────────┤
│                                         │
│  1. 显式生命周期                        │
│     语法: 'a, 'b, 'static               │
│     用途: 函数签名、结构体定义            │
│     示例: fn foo<'a>(x: &'a i32)         │
│                                         │
│  2. 隐式生命周期 (省略规则)              │
│     规则 1: 每个引用参数获得独立生命周期   │
│     规则 2: 只有一个输入生命周期时，      │
│             输出使用相同生命周期          │
│     规则 3: 多个输入时，输出生命周期      │
│             需要显式标注                  │
│                                         │
│  3. 'static 生命周期                    │
│     含义: 整个程序运行期间有效            │
│     示例: 字符串字面量 "hello"            │
│     注意: 不意味着变量是静态的            │
│                                         │
└─────────────────────────────────────────┘
```

### 5.3 生命周期省略规则详解

```rust
// 规则 1: 每个引用参数获得独立生命周期
fn foo(x: &i32)           →  fn foo<'a>(x: &'a i32)
fn bar(x: &i32, y: &i32)  →  fn bar<'a, 'b>(x: &'a i32, y: &'b i32)

// 规则 2: 只有一个输入时，输出相同
fn foo(x: &i32) -> &i32   →  fn foo<'a>(x: &'a i32) -> &'a i32

// 规则 3: &self 时，输出使用 &self 的生命周期
fn method(&self) -> &T    →  fn method<'a>(&'a self) -> &'a T

// 需要显式标注的情况
fn longest(x: &str, y: &str) -> &str  // 错误！无法确定返回哪个
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str  // 正确
```

### 5.4 生命周期子类型关系

```text
长生命周期是短生命周期的子类型。

关系: 'static <: 'a <: 'b (如果 'a 比 'b 长)

含义: 需要短生命周期的地方可以使用长生命周期
      (长生命周期保证至少活到短生命周期结束)

示例:
  fn foo<'a>(x: &'a str)

  可以传入: &"hello" (&'static str)
  因为: 'static <: 'a
```

## 第六部分: 类型系统语义

### 6.1 类型作为约束

```text
类型 = 值的集合 + 允许的操作

i32: {..., -1, 0, 1, ...}
     操作: +, -, *, /, %

bool: {true, false}
      操作: &&, ||, !

String: {所有有效的 UTF-8 字符串}
        操作: push, pop, len, etc.
        约束: 拥有堆内存，不能 Copy
```

### 6.2 Trait 系统的逻辑

```text
Trait = 接口 (类型必须实现的方法)

逻辑关系:
  类型 T 实现 Trait A  ⇔  T 拥有 A 要求的所有方法

泛型约束:
  fn foo<T: Trait>(x: T)  ⇔  对于所有实现 Trait 的 T

多 Trait 约束:
  T: TraitA + TraitB  ⇔  T 同时实现 TraitA 和 TraitB

关联类型:
  trait Iterator { type Item; }
  每个实现者决定 Item 的具体类型
```

### 6.3 泛型的单态化

```text
编译时行为:
  fn identity<T>(x: T) -> T { x }

  identity(5i32)     →  生成 identity_i32
  identity("hello")  →  生成 identity_str

运行时行为:
  每个单态化版本是独立的函数
  没有运行时类型检查的开销
```

## 第七部分: 内存模型与操作语义

### 7.1 Rust 的内存模型层次

```text
┌─────────────────────────────────────┐
│        抽象内存模型                  │
│   (变量、值、所有权)                 │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│        逻辑内存模型                  │
│   (栈、堆、引用、生命周期)            │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│        物理内存模型                  │
│   (地址、字节、对齐、布局)            │
└─────────────────────────────────────┘
```

### 7.2 栈与堆的语义区别

```text
栈 (Stack):
  - 自动管理 (LIFO)
  - 固定大小 (编译时已知)
  - 快速分配/释放
  - 存储: 基本类型、固定大小数组、引用

堆 (Heap):
  - 手动管理 (通过所有权)
  - 动态大小 (运行时确定)
  - 相对慢速分配/释放
  - 存储: String, Vec, Box, 动态数据

关键区别:
  栈数据: Copy trait (隐式复制)
  堆数据: Move semantics (所有权转移)
```

### 7.3 操作语义的两层抽象

#### 大步语义 (Big-Step)

```text
σ; h ⊢ e ⇓ v; h'

含义: 在环境 σ 和堆 h 下，表达式 e 求值为 v，产生新堆 h'

特点:
  - 描述最终结果
  - 适合类型保持证明
  - 不显示中间步骤
```

#### 小步语义 (Small-Step)

```text
⟨σ, h, e⟩ → ⟨σ', h', e'⟩

含义: 表达式 e 单步求值为 e'，状态相应更新

特点:
  - 显示求值过程
  - 适合并发和步数分析
  - 可以观察中间状态
```

### 9.2 详细示例: 借用冲突检测

```rust
// 示例 1: 合法的多重不可变借用
let x = 5;
let y = &x;      // 第一个 &x
let z = &x;      // 第二个 &x (合法，都是读取)
println!("{} {}", y, z);

// 示例 2: 非法的混合借用
let mut x = 5;
let y = &x;      // 不可变借用
let z = &mut x;  // 错误! 不能同时有 & 和 &mut
// 原因: y 可能被用来读取，而 z 可以修改，导致数据竞争

// 示例 3: 合法的可变借用序列
let mut x = 5;
let y = &mut x;  // 第一个 &mut x
*y = 10;
// y 在这里之后不再使用
let z = &mut x;  // 合法，y 已经失效
*z = 20;

// 示例 4: 生命周期冲突
let y: &i32;
{
    let x = 5;
    y = &x;      // 错误! x 的生命周期太短
    // x 在这里被释放，但 y 仍然有效
}
// y 在这里无效，但编译器不允许这种情况
```

```coq
Example ex_immutable_borrow : forall Δ Θ,

  let e := EBorrow RStatic Shrd (PVar "x") in
  has_type Δ Γ Θ e (TRef RStatic Shrd ti32).
Proof.
  intros.
  apply T_Borrow.
  - apply PT_Var. reflexivity.  (* x 有类型 i32 *)
  - apply OS_Base. reflexivity. (* x 访问安全 *)

Qed.

Proof.
  (* 证明: 当 x 被可变借用时，不能再有不可变借用 *)
```

## 第十部分: 反例与边界情况

### 10.1 常见错误模式

```text
错误模式 1: 使用已移动的值
  let s = String::from("hello");
  let s2 = s;
  println!("{}", s);  // 错误! s 已被移动

  原因: 所有权已转移给 s2
  解决: 使用 clone() 或引用

错误模式 2: 悬垂引用
  let y: &i32;
  {
      let x = 5;
      y = &x;
  }  // x 在这里释放
  println!("{}", y);  // 错误! y 指向无效内存

  原因: 引用的生命周期长于被引用者
  解决: 确保引用不超出被引用者的作用域

错误模式 3: 同时有可变和不可变借用
  let mut x = 5;
  let y = &x;
  let z = &mut x;  // 错误!

  原因: 可能导致数据竞争
  解决: 分开借用或使用内部可变性

错误模式 4: 在循环中创建引用
  let mut data = vec![1, 2, 3];
  for i in &mut data {
      data.push(*i);  // 错误!
  }

  原因: 迭代期间不能修改集合
  解决: 收集修改后再应用，或使用索引
```

### 10.2 边界情况分析

```text
边界 1: 自引用结构体
  struct SelfRef<'a> {
      data: String,
      ptr: &'a str,  // 指向 data
  }

  问题: 移动结构体后，ptr 仍然指向旧的 data 位置
  解决: 使用 Pin 或重构设计

边界 2: 协变/逆变/不变
  - &T 对 T 是协变的
  - &mut T 对 T 是不变的
  - Box<T> 对 T 是协变的

  原因: 防止类型系统的声音性漏洞

边界 3: 生命周期省略失败
  fn longest(x: &str, y: &str) -> &str  // 编译错误

  原因: 编译器无法确定返回值的生命周期
  解决: 显式标注生命周期
```

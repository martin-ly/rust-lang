# 定理分级规范（Theorem Tier Specification）
>
> **EN**: Theorem Tier Spec
> **Summary**: Theorem Tier Spec. Core Rust concept.
> **定位**: 统一所有 `concept/` 文件中"定理/引理/推论"的可信度标注标准。
> **原则**: 消除"伪形式化"——定理编号必须有对应的分级，证明深度必须与分级匹配。
> **适用范围**: `concept/` 全部 45 个文件，retroactive 标注 + 新增内容强制执行。

>
> **来源**: [Rust Reference](https://doc.rust-lang.org/reference/)
> **前置概念**: N/A
> **后置概念**: N/A

## 一、三级定理体系

### Tier 1 — 外部验证定理（External Theorem）

**定义**: 有同行评审的学术论文、官方规范或形式化验证工具给出的完整证明。

**标注格式**:

```markdown
> **定理 T-XXX** [Tier 1] — 定理名称
>
> **陈述**: ...
> **证明**: 见 [来源: RB18 — §3.2] 或 [来源: REF — §4.1.9]。此处仅给出**直觉概述**。
> **直觉概述**: ...
```

**适用场景**:

- RustBelt (POPL 2018) 证明的 Safe Rust 内存安全（Memory Safety）定理
- Ralf Jung 论文中证明的 Stacked Borrows / Tree Borrows 操作语义性质
- Pierce TAPL 中给出的类型系统（Type System）元定理（Progress + Preservation）
- Wadler 1989 参数性定理的形式化陈述

**判断标准**:

- [ ] 存在完整的、可检查的数学证明（非直觉论证）
- [ ] 已发表于同行评审的会议/期刊，或被官方规范采纳
- [ ] 证明不依赖于"显然"或"直观上"等模糊词汇

---

### Tier 2 — 证明草图定理（Proof Sketch）

**定义**: 基于已建立的形式化框架，给出了完整的证明结构，但省略了部分技术性细节（如归纳步骤的机械展开）。

**标注格式**:

```markdown
> **定理 T-XXX** [Tier 2] — 定理名称
> **陈述**: ...
> **证明草图**:
> 1. ...（建立不变量）
> 2. ...（归纳步骤，见 [来源: TAPL — 引理 9.3.1]）
> 3. ...（结论）
> **省略细节**: 步骤 2 的完整归纳假设展开见 [来源: 原始论文附录]。
> **形式化根基**: [来源: RB18] / [来源: TAPL]
```

**适用场景**:

- 基于 RustBelt 框架推导的特定性质（但未在原始论文中显式证明）
- 基于分离逻辑推导的 Rust 特定模式的安全性
- 从 System F 规则推导 Rust 泛型（Generics）系统的类型保持性

**判断标准**:

- [ ] 有明确的证明结构（引理 → 定理）
- [ ] 每个步骤都有形式化依据（引用（Reference）外部定理或公理）
- [ ] 省略的细节是"机械的"而非"关键的"
- [ ] 明确标注了省略的内容和补充来源

---

### Tier 3 — 直觉命题（Intuitive Proposition）

**定义**: 基于工程直觉、编译器行为观察或逻辑推理得出的论断，**无完整形式化证明**。

**标注格式**:

```markdown
> **命题 P-XXX** [Tier 3] — 命题名称
> **陈述**: ...
> **论证**（非证明）:
> - ...（编译器行为观察）
> - ...（反例验证）
> - ...（与其他已知性质的类比）
> **局限性**: 此命题基于以下假设...，在以下场景可能不成立...
> **待形式化**: 目前无已知的完整形式化证明。可能的证明方向：...
```

**关键区别**: Tier 3 **不使用"定理"一词**，改用"命题"或"论断"。

**适用场景**:

- "Safe Rust 无内存泄漏（模循环引用（Reference））"——Rust 不保证无泄漏，此命题本身就有语义边界
- "单态化（Monomorphization） ⟹ 零成本抽象（Zero-Cost Abstraction） ⟹ 语义保持"——LLVM 优化差异使"同构"断言过强
- "Vec 扩容因子约为 1.5–2×"——实现细节可能随版本变化
- 任何带有 💡 原创分析 标注的"定理"

**判断标准**:

- [ ] 无外部形式化证明支撑
- [ ] 基于编译器行为观察、测试验证或工程推理
- [ ] 明确标注了假设条件和失效场景
- [ ] 不使用 "∴"、"QED"、"得证" 等形式化证明结语

## 二、Retroactive 标注执行指南

### 第一步：识别现有"定理"

在每个概念文件中搜索以下模式：

- `**定理` 或 `> **定理`
- `T-` 编号系统
- `⟹` 推理链
- `QED` 或 `得证`

### 第二步：分级判定

对每个"定理"回答以下问题：

1. 是否有外部论文/规范给出了完整证明？ → **Tier 1**
2. 是否基于外部框架给出了完整的证明结构（即使省略细节）？ → **Tier 2**
3. 是否仅基于直觉、观察或原创推理？ → **Tier 3**（改为"命题"）

### 第三步：格式更新

根据分级替换原文：

| 原格式 | Tier 1 | Tier 2 | Tier 3 |
|:---|:---|:---|:---|
| 标题 | `**定理 T-001** [Tier 1]` | `**定理 T-001** [Tier 2]` | `**命题 P-001** [Tier 3]` |
| 证明措辞 | "证明：...（概述）" | "证明草图：..." | "论证：..." |
| 结语 | 引用外部证明 | 标注省略细节 | 标注局限性和失效条件 |
| 来源要求 | 必须引用具体论文章节 | 必须引用形式化框架 | 必须引用观察依据或标注 💡 |

### 第四步：一致性检查

- 文件内所有 Tier 1/2/3 标注格式统一
- 无"伪形式化"残留（如 Tier 3 使用 "∴" 或 "QED"）
- 定理一致性（Coherence）矩阵中的每个条目都有对应的 Tier 标注

## 三、示例转换

### 示例 1：Tier 1 转换

**原文**（`01_ownership.md`）:

```markdown
> **定理 T-001**: Safe Rust 无内存泄漏（模循环引用）
> **证明**: ① 所有权唯一性保证每块堆内存有唯一的所有者... ② 当所有者离开作用域，Drop 被调用... ③ 循环引用通过 Rc/Arc 打破，但 Rust 不保证无泄漏...
```

**问题**: 三句话的直觉推理，无外部证明支撑。

**转换后**:

```markdown
> **命题 P-001** [Tier 3] — Safe Rust 无内存泄漏（模循环引用）
> **陈述**: 在 Safe Rust 中，无循环引用的代码不会发生堆内存泄漏。
> **论证**:
> - 所有权唯一性保证每块堆内存有唯一的所有者（`Box<T>`、`Vec<T>`）
> - 当所有者离开作用域，编译器插入 `drop` 调用，释放内存
> - `Rc`/`Arc` 循环引用确实可导致泄漏，但 Rust 的设计哲学是"不保证无泄漏，只保证无 UAF/double-free"
> **局限性**:
> - 此命题**不适用于** `Rc`/`Arc` 循环引用场景
> - `std::mem::forget` 可人为阻止 Drop
> - RustBelt 证明的是**内存安全**（无 UAF），而非**资源无泄漏**
> **待形式化**: 资源无泄漏的形式化定义需要线性逻辑（非仿射逻辑），见 [来源: LL87]。
```

### 示例 2：Tier 2 转换

**原文**（`02_generics.md`）:

```markdown
> **定理 T-003**: 单态化 ⟹ 零成本抽象 ⟹ 语义保持
> **证明草图**: 对泛型函数 `f<T>` 的每个单态化实例 `f_Concrete`，编译器生成与手写 `f_concrete` 逐指令同构的代码...
```

**问题**: "逐指令同构"断言过强（LLVM 优化差异）。

**转换后**:

```markdown
> **定理 T-003** [Tier 2] — 单态化保持语义等价性
> **陈述**: 在单态化过程中，泛型函数的每个实例在 MIR 层面与对应的手写单型函数语义等价。
> **证明草图**:
> 1. Rust 单态化在 MIR → LLVM IR 之前发生，替换类型参数为具体类型
> 2. 替换后的 MIR 与手写单型函数的 MIR 结构相同（归纳于类型结构）
> 3. 后续 LLVM 优化可能引入差异，但** observable behavior **保持不变（由 LLVM 的正确性保证）
> **省略细节**: 步骤 2 的完整归纳证明需要形式化 MIR 语义，目前无公开的 Rust MIR 形式化规范。
> **局限性**:
> - "零成本"指**运行时零成本**，编译时间和二进制大小可能增加
> - LLVM 的内联决策可能与手写代码不同，导致最终机器码差异
> **形式化根基**: 来源: [Rust Reference — §6.1](https://doc.rust-lang.org/reference/) 单态化语义描述（非形式化证明）。
```

## 四、自动化校验规则

### 脚本检查点

```python
# pseudocode
def check_theorem_tier(file_path):
    content = read(file_path)
    theorems = extract_theorems(content)

    for t in theorems:
        # 规则 1: 必须有 Tier 标注
        assert "[Tier" in t.header, f"{file_path}: {t.id} 缺少 Tier 标注"

        # 规则 2: Tier 3 不能使用 "定理"
        if t.tier == 3:
            assert "定理" not in t.header, f"{file_path}: {t.id} Tier 3 应使用'命题'"

        # 规则 3: Tier 1/2 不能只有直觉论证
        if t.tier in [1, 2]:
            assert len(t.references) > 0, f"{file_path}: {t.id} Tier 1/2 必须有外部引用"

        # 规则 4: 禁止伪形式化结语
        forbidden = ["∴", "QED", "得证", "证毕"]
        if t.tier == 3:
            for word in forbidden:
                assert word not in t.body, f"{file_path}: {t.id} Tier 3 不能使用 '{word}'"
```

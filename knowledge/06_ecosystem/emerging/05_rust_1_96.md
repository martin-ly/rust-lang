## 🎯 版本概览

Rust 1.96 主要聚焦于：

- **标准库扩展**: `assert_matches!`、`core::range::*` 类型族、`NonZero` 范围迭代、`From<T>`  for `LazyLock/LazyCell/AssertUnwindSafe`
- **语言级改进**: `expr` metavariable 传递给 `cfg`、Never 类型在 tuple 中强制转换、`ManuallyDrop` 常量模式
- **Cargo 安全**: 修复 CVE-2026-5222 和 CVE-2026-5223
- **平台支持**: LoongArch link relaxation、s390x vector registers、riscv64gc Fuchsia RVA22

## 🚀 语言特性 (Language)

### 1. `expr` Metavariable 传递给 `cfg`

宏中可以将 `expr` 类型的 metavariable 传递给 `cfg` 属性：

```rust
macro_rules! configurable {
    ($e:expr) => {
        #[cfg($e)]
        fn enabled() {}
    };
}
```

> **来源**: [Rust Reference — Macros](https://doc.rust-lang.org/reference/macros-by-example.html)

### 2. Never 类型在 Tuple 中强制转换

`!`（never type）在 tuple 表达式中始终被强制转换：

```rust
fn never_returns() -> ! {
    panic!("never")
}

let tuple: (i32, i32) = (never_returns(), 42);
// 1.96 之前：可能需要显式类型标注
// 1.96+：编译器自动将 `!` coerce 为目标类型
```

### 3. `ManuallyDrop` 常量作为模式

允许在模式匹配中使用 `ManuallyDrop` 类型的常量（修复 1.94.0 引入的回归）：

```rust
use std::mem::ManuallyDrop;

const FLAG: ManuallyDrop<u32> = ManuallyDrop::new(1);

match ManuallyDrop::new(1) {
    FLAG => println!("matched constant"),
    _ => println!("other"),
}
```

### 4. s390x Vector Registers 内联汇编支持

`s390x` 架构现在支持向量寄存器内联汇编。

```rust
#[cfg(target_arch = "s390x")]
unsafe fn vector_add(a: &[i32; 4], b: &[i32; 4]) -> [i32; 4] {
    let mut result: [i32; 4] = [0; 4];
    std::arch::asm!(
        "vl %v0, 0({0})",
        "vl %v1, 0({1})",
        "vaf %v2, %v0, %v1",
        "vst %v2, 0({2})",
        in(reg) a.as_ptr(),
        in(reg) b.as_ptr(),
        in(reg) result.as_mut_ptr(),
        options(nostack)
    );
    result
}
```

> **注意**: s390x 向量寄存器支持需要对应的硬件和 LLVM 后端支持。标准库目前尚未暴露高层向量 API，需通过 `std::arch::asm!` 直接调用。

## 🔧 编译器改进 (Compiler)

### 1. LoongArch Linux 目标启用 Link Relaxation

LoongArch Linux 目标（如 `loongarch64-unknown-linux-gnu`）现已启用链接时 relaxation 特性。这允许链接器在最终链接阶段优化指令序列，将部分长跳转/长加载指令松弛为更短、更高效的版本，从而减少二进制体积并提升运行时性能。

> **影响**: 仅影响 LoongArch 架构的 Linux 目标，其他平台无变化。

### 2. `riscv64gc-unknown-fuchsia` 基线升级至 RVA22

RISC-V Fuchsia 目标的 ISA 基线从旧版本升级至 **RVA22**（RISC-V Application Profile 2022），并启用了向量扩展支持。这意味着编译器将假设目标设备至少支持 RVA22 规范中定义的所有强制指令集特性。

> **迁移注意**: 如果你的 Fuchsia RISC-V 设备硬件较旧，可能需要通过 `-Ctarget-feature` 显式关闭向量扩展，或降级至自定义目标规格。

## 📦 标准库新特性 (Libraries)

### 5. `assert_matches!` / `debug_assert_matches!`

```rust,ignore
use std::assert_matches;

let result: Result<i32, &str> = Ok(42);
assert_matches!(result, Ok(n) if n > 0);

debug_assert_matches!(config, Some(Config::Debug));
```

**与 `assert!(matches!(...))` 的区别**:

- 错误信息更友好（显示实际值 vs 模式）
- 支持 guard 条件（`if expr`）
- 绑定不可导出到宏外部（如需提取值，使用 `if let`）

> **版本勘误**: 此前部分文档错误标注为 "1.77 稳定"。实际稳定版本为 **1.96.0**。

### 6. `core::range` 类型族

Rust 1.96.0 补齐了 `core::range` 模块的核心类型：

| 类型 | 对应 `std::ops` | 迭代器类型 | 含义 |
|:---|:---|:---|:---|
| `core::range::Range` | `std::ops::Range` | `RangeIter` | 半开区间 `[start, end)` |
| `core::range::RangeFrom` | `std::ops::RangeFrom` | `RangeFromIter` | 无限区间 `[start, ∞)` |
| `core::range::RangeToInclusive` | `std::ops::RangeToInclusive` | `RangeToInclusiveIter` | 闭区间 `(-∞, end]` |
| `core::range::RangeInclusive` (1.96.0) | `std::ops::RangeInclusive` | `RangeInclusiveIter` | 闭区间 `[start, end]` |

```rust
use core::range::{Range, RangeFrom, RangeToInclusive};

let r = Range { start: 1, end: 5 };
let sum: i32 = r.into_iter().sum();
assert_eq!(sum, 10);

let rf = RangeFrom { start: 10 };
let tenth: i32 = rf.into_iter().nth(10).unwrap();
assert_eq!(tenth, 20);
```

### 7. `From<T>` for `LazyCell<T, F>` / `LazyLock<T, F>` / `AssertUnwindSafe<T>`

```rust
use std::sync::LazyLock;
use std::cell::LazyCell;
use std::panic::AssertUnwindSafe;

// 直接从值构造，无需闭包
let lazy: LazyLock<String> = LazyLock::from("hello".to_string());
let cell: LazyCell<String> = LazyCell::from("world".to_string());
let safe: AssertUnwindSafe<i32> = AssertUnwindSafe::from(42);
```

### 8. `NonZero` 整数范围迭代

```rust
use std::num::NonZeroU32;
use std::ops::Range;

let start = NonZeroU32::new(1).unwrap();
let end = NonZeroU32::new(5).unwrap();
let range: Range<NonZeroU32> = start..end;

let values: Vec<u32> = range.map(|nz| nz.get()).collect();
assert_eq!(values, vec![1, 2, 3, 4]);
```

### 9. "Valid for Read/Write" 定义重构

Rust 标准库对指针"有效可读/可写"（valid for read/write）的内存模型定义进行了调整：**空指针（null pointer）现在默认不被视为 valid**。此前某些文档和内部实现将 null 视为 trivially valid（因为无法通过它读取或写入），这导致了 `NonNull` 与原始指针之间的不一致。

这一变更为纯文档和概念层面的澄清，实际编译的代码行为未变。各个方法（如 `ptr::read`、`ptr::write`）的文档现在单独注明 null 是否允许。

> **影响**: 无运行时行为变更。如果你编写了依赖"null 是 valid pointer"这一不准确假设的 unsafe 代码，建议重新审视。

### 10. SGX 延迟主机地址解析修复

修复了在 Intel SGX 可信执行环境中，通过 `ToSocketAddrs` 进行主机名查找时可能出现的延迟问题。此前在 SGX 环境下，某些 DNS 解析请求会被错误地延迟到运行时，而不是在连接建立时立即失败。

> **影响**: 仅影响使用 SGX 目标（`x86_64-fortanix-unknown-sgx`）的网络应用程序。

## 📖 Rustdoc 改进

### 1. Deprecation Notes 渲染规范化

`#[deprecated]` 属性的说明文本现在与普通文档一样渲染。此前 Rustdoc 对弃用说明使用了 `white-space: pre-wrap` CSS 属性，并会剥离渲染后 HTML 中的 `<p>` 元素，这导致多行弃用说明的行为不直观。

**新行为**:

- 弃用说明中的 Markdown 被正常解析和渲染
- 标准 Markdown 换行规则适用（如需强制换行，使用两个空格加换行符）

```rust
#[deprecated(
    since = "1.96.0",
    note = "This function is deprecated.  \nPlease use `new_function()` instead."
)]
pub fn old_function() {}
```

### 2. `missing_doc_code_examples` Lint 范围缩小

`missing_doc_code_examples` lint 不再对 **impl items**（即 `impl` 块中的方法实现）发出警告。此前该 lint 会对 trait 实现中的每个方法都要求代码示例，造成大量不必要的噪音。

```rust,ignore
/// Trait 文档中的代码示例仍被检查
pub trait MyTrait {
    /// 需要代码示例
    fn trait_method();
}

impl MyTrait for MyStruct {
    // 1.96+：此处不再需要代码示例
    fn trait_method() {}
}
```

### 3. 侧边栏中方法与关联函数分离

Rustdoc 生成的文档侧边栏现在将 **方法**（`self` 接收者）和 **关联函数**（无 `self` 接收者）分为两个独立的分组显示。这使得导航大型类型时更容易区分构造函数/工具函数与实例方法。

## 🛠️ Cargo 变更

### 9. Git + Alternate Registry 共存

允许同一个依赖同时指定 git 仓库和 alternate registry：

```toml
[dependencies]
my-crate = { git = "https://github.com/example/my-crate", registry = "my-registry" }
```

本地开发使用 git 版本，发布时使用 registry 版本。

### 10. `target.'cfg(..)'.rustdocflags` 配置支持

```toml
[target.'cfg(unix)']
rustdocflags = ["--cfg", "docsrs"]
```

### 11. 安全修复

- **CVE-2026-5222**: Cargo sparse registry URL 规范化错误（`.git` 后缀剥离，影响 1.68–1.95）
- **CVE-2026-5223**: Cargo tarball symlink 缓存覆盖（crates.io 不受影响，第三方注册表有风险）

### 11. CVE 安全修复详情与缓解

**CVE-2026-5222**: Cargo sparse registry URL 规范化错误

- **影响范围**: Cargo 1.68 至 1.95
- **漏洞**: 稀疏注册表索引的 `.git` 后缀剥离逻辑错误，可能导致依赖解析时访问错误的注册表 URL
- **缓解**: 升级至 1.96.0+。如果暂时无法升级，避免使用非标准路径的稀疏注册表。

**CVE-2026-5223**: Cargo tarball symlink 缓存覆盖

- **影响范围**: 使用第三方或私有注册表的用户
- **漏洞**: 恶意构造的 crate tarball 中的符号链接可能覆盖 Cargo 缓存目录外的文件
- **缓解**: 升级至 1.96.0+。crates.io 已部署服务端缓解措施，不受影响。

## 🧩 Internal Changes

以下变更不影响 Rust 的公共接口，但代表了 rustc 及相关工具在性能或内部实现上的显著改进。

### 1. JSON Target 规格: aarch64 Softfloat 要求

对于 aarch64 softfloat 目标（如 `aarch64-unknown-none-softfloat`），JSON 目标规格现在**必须**显式设置 `rustc_abi` 字段为 `"softfloat"`。此前该字段为可选，缺失时可能导致 ABI 不匹配的边缘情况。

```json
{
    "llvm-target": "aarch64-unknown-none",
    "target-endian": "little",
    "target-pointer-width": "64",
    "arch": "aarch64",
    "rustc_abi": "softfloat"
}
```

### 2. Target Specs: 更严格的 LLVM ABI 检查

目标规格中的 LLVM ABI 字符串现在经过更严格的验证，且必须与 `cfg(target_abi)` 正确关联。这防止了自定义目标规格中因 ABI 字符串拼写错误或 LLVM 版本不匹配导致的模糊编译错误。

> **影响范围**: 仅影响使用自定义 JSON target spec 的用户，标准目标不受影响。

## ⚠️ 兼容性注意 (Compatibility Notes)

### 1. `#[repr(Int)]` Enum Layout 修复

修复了涉及 uninhabited zero-sized types (ZST) 字段时 `#[repr(Int)]` 枚举的边缘情况布局错误。此前某些包含不可构造 ZST 字段的枚举在内存布局上不符合 C 兼容承诺。

```rust
#[repr(u8)]
enum Example {
    A,
    B(std::convert::Infallible), // uninhabited ZST
}
```

> **迁移建议**: 如果你依赖此类枚举的精确内存布局（例如用于 FFI 或 unsafe 指针运算），请重新验证布局是否符合预期。可使用 `std::mem::size_of` 和 `offset_of!` 宏进行检查。

### 2. 禁止 Unsize-Coerce 到非 `Deref` 的 `Pin<Foo>`

此前编译器错误地允许将类型 unsize-coerce 为 `Pin<Foo>`，即使 `Foo` 未实现 `Deref`。这会产生一个几乎没有可用公共 API 的类型，属于编译器漏洞。

```rust
// 1.96+：以下代码现在报错
// let p: Pin<&dyn Trait> = Pin::new(&Concrete);
// 错误：无法 unsize-coerce，因为 `&Concrete` 不满足 `Deref`
```

### 3. `#![reexport_test_harness_main]` 被正确 Gate

`#![reexport_test_harness_main]` 属性此前意外地被稳定，现已正确限制为 nightly only。该属性允许自定义测试运行时的入口函数名称，属于高级不稳定特性。

> **影响**: 如果你在 stable 通道使用了此属性，1.96 起将收到编译错误。解决方案是移除该属性，使用默认测试入口。

### 4. RPITIT 类型隐私过严报错

返回位置 impl trait in traits (RPITIT) 的实现现在会在其返回类型过于私有时报错。此前某些情况下，返回类型引用了私有类型但未触发隐私错误，导致生成的文档与实际 API 不符。

### 5. `uninhabited_static` Lint 升级为 Deny-by-Default

`uninhabited_static` lint 现在在依赖项中也会被报告，且默认等级从 `warn` 提升至 `deny`。此 lint 检测 `static` 变量类型为 uninhabited（不可实例化）的情况，这通常意味着代码逻辑错误。

```rust,ignore
// 现在产生 deny 级错误
static NEVER: std::convert::Infallible = todo!();
```

### 6. windows-gnu 使用非分割 Debuginfo

Windows GNU 目标的发行版现在包含非分割 debuginfo。这改善了崩溃时的 backtrace 质量。注意：**这不改变** `rustc`/`cargo` 的默认输出行为，仅影响通过 `rustup` 分发的预构建编译器二进制。

### 7. 移除 `-Csoft-float` 选项

`-Csoft-float` 编译器选项已被移除。该选项此前用于强制软件浮点运算，但实现不完善且与多个目标冲突。请改用 `target-feature` 控制硬件浮点支持：

```bash
# 旧（已移除）
rustc -Csoft-float

# 新
rustc --target=my-target.json -Ctarget-feature=-fp-armv8
```

### 8. `use S::{self as Other}` 被禁止

导入结构体时使用 `::{self}` 语法要求被导入项具有模块父级。结构体本身不是模块，因此 `use S::{self as Other}` 现在被禁止。

```rust
struct S;

// 1.96+：错误
// use S::{self as Other};

// 正确做法
use S as Other;
```

### 9. 重复属性首者优先

对于 `export_name`、`link_name` 和 `link_section` 属性，当同一项上存在多个同名属性时，**第一个**属性现在优先。此前行为未定义，某些情况下最后一个优先。

```rust,ignore
#[export_name = "first"]  // 生效
#[export_name = "second"] // 被忽略（1.96 起）
pub fn foo() {}
```

### 10. 最低外部 LLVM 升至 21

从源码构建 rustc 所需的最低外部 LLVM 版本从 20 提升至 21。这不影响通过 `rustup` 使用预构建编译器的用户。

### 11. AVR 目标 `c_double` 改为 `f32`

AVR 架构上 C 语言的 `double` 类型默认是 32 位。Rust 的 `libc::c_double` 现已同步改为 `f32`（而非 `f64`）以匹配 C 的行为。这是一个破坏性变更，但对于 AVR 嵌入式开发者而言是必要的正确性修复。

```rust,ignore
// AVR 目标上
let d: libc::c_double = 1.0f32; // 正确
// let d: libc::c_double = 1.0f64; // 错误：类型不匹配
```

## ✅ 迁移检查清单

在将项目升级至 Rust 1.96.0 之前，建议完成以下检查：

- [ ] 运行 `cargo test` 确保现有测试通过
- [ ] 检查是否使用了 `#![reexport_test_harness_main]`（stable 项目需移除）
- [ ] 检查是否有 `use S::{self as Other}` 语法（需改为 `use S as Other`）
- [ ] 检查是否依赖 `-Csoft-float`（需改用 target-feature）
- [ ] 对于 AVR 项目：确认 `libc::c_double` 的使用已适配 `f32`
- [ ] 对于 LoongArch 项目：确认链接脚本与 link relaxation 兼容
- [ ] 对于自定义 JSON target spec：确认 `rustc_abi` 和 LLVM ABI 字段正确
- [ ] 对于 Fuchsia RISC-V：确认硬件支持 RVA22 + vector
- [ ] 运行 `cargo doc` 检查弃用说明渲染是否符合预期

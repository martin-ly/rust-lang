# 度量与测量指南

## 概述

本文档提供Rust安全关键系统开发中的关键度量指标定义、测量方法和目标值，支持项目管理和质量评估。

---

## 1. 代码质量度量

### 1.1 代码复杂度

| 度量 | 定义 | ASIL D目标 | 工具 |
|------|------|-----------|------|
| **圈复杂度** | 线性独立路径数 | ≤ 10 | clippy |
| **认知复杂度** | 理解难度 | ≤ 15 | 手动 |
| **嵌套深度** | 最大嵌套层级 | ≤ 3 | clippy |
| **函数长度** | 代码行数 | ≤ 50 | clippy |
| **参数数量** | 函数参数个数 | ≤ 5 | clippy |

```rust
// 圈复杂度示例
fn calculate(a: i32, b: i32, op: Op) -> i32 {
    match op {           // +1
        Op::Add => a + b,
        Op::Sub => a - b,
        Op::Mul => a * b,
        Op::Div => {     // +1
            if b != 0 {  // +1
                a / b
            } else {     // +1
                0
            }
        }
    }
}
// 总圈复杂度: 4
```

### 1.2 代码规模

```
度量指标:
├── 物理行数 (LOC)
│   ├── 定义: 包含代码的行数
│   └── 目标: 无限制，但需模块化
│
├── 逻辑行数 (LLOC)
│   ├── 定义: 语句数量
│   └── 目标: 函数≤30语句
│
├── 注释密度
│   ├── 定义: 注释行/代码行
│   └── 目标: 20-40%
│
└── 空行密度
    ├── 定义: 空行/总行数
    └── 目标: 10-20%
```

## 2. 测试覆盖率度量

### 2.1 覆盖率类型

| 类型 | 定义 | ASIL D | ASIL C | 工具 |
|------|------|--------|--------|------|
| **语句覆盖** | 执行语句比例 | 100% | 100% | llvm-cov |
| **分支覆盖** | 执行分支比例 | 100% | 100% | llvm-cov |
| **MC/DC** | 修改条件决策覆盖 | 100% | - | llvm-cov |
| **函数覆盖** | 调用函数比例 | 100% | 100% | llvm-cov |
| **行覆盖** | 执行代码行比例 | 100% | 100% | tarpaulin |

### 2.2 覆盖率测量

```bash
# 使用cargo-llvm-cov
cargo llvm-cov --all-features --lcov --output-path lcov.info

# 生成HTML报告
cargo llvm-cov --all-features --html

# 仅测试特定模块
cargo llvm-cov -p my-crate --lcov
```

### 2.3 覆盖率检查表

```rust
#[cfg(test)]
mod coverage {
    /// MC/DC测试示例
    #[test]
    fn test_mcdc_condition_a() {
        // 条件A为真
        assert!(complex_condition(true, false, false));

        // 条件A为假，B和C保持相同
        assert!(!complex_condition(false, false, false));
    }

    #[test]
    fn test_mcdc_condition_b() {
        // 条件B为真
        assert!(complex_condition(false, true, false));

        // 条件B为假，A和C保持相同
        assert!(!complex_condition(false, false, false));
    }

    fn complex_condition(a: bool, b: bool, c: bool) -> bool {
        a || (b && c)
    }
}
```

## 3. 缺陷度量

### 3.1 缺陷密度

| 阶段 | 度量 | 目标 | 计算方法 |
|------|------|------|----------|
| **开发** | 编译错误/KLOC | < 10 | rustc错误计数 |
| **静态分析** | 警告/KLOC | < 5 | Clippy警告 |
| **测试** | 缺陷/KLOC | < 1 | 发现缺陷数 |
| **现场** | 缺陷/KLOC | < 0.1 | 现场报告 |

### 3.2 缺陷分类

```
严重程度:
├── 致命: 系统崩溃/安全违规
├── 严重: 主要功能失效
├── 中等: 次要功能问题
├── 轻微: 界面/文档问题
└── 建议: 改进建议

缺陷类型:
├── 需求缺陷
├── 设计缺陷
├── 编码缺陷
├── 文档缺陷
└── 流程缺陷
```

## 4. 过程度量

### 4.1 开发效率

| 度量 | 定义 | 目标 | 收集方法 |
|------|------|------|----------|
| **生产率** | LOC/人月 | 视复杂度 | 代码统计 |
| **缺陷率** | 缺陷/人月 | < 5 | 缺陷跟踪 |
| **返工率** | 返工工时/总工时 | < 10% | 工时记录 |
| **评审效率** | 缺陷/评审小时 | > 2 | 评审记录 |

### 4.2 时间度量

```
周期时间:
├── 需求到设计: 2-4周
├── 设计到编码: 4-8周
├── 编码到测试: 2-4周
├── 测试到发布: 4-12周
└── 总周期: 视项目规模

响应时间:
├── 关键缺陷: 24小时
├── 严重缺陷: 1周
├── 中等缺陷: 2周
└── 轻微缺陷: 下次迭代
```

## 8. 度量工具

### 8.1 自动收集

```yaml
# GitHub Actions度量收集
- name: Collect Metrics
  run: |
    # 代码行数
    echo "LOC=$(find src -name '*.rs' | xargs wc -l | tail -1)" >> $GITHUB_ENV

    # 测试覆盖
    cargo llvm-cov --json > coverage.json

    # 复杂度
    cargo clippy --message-format=json | jq '.complexity'
```

# Q4 2026 I18N 工具决策备忘录：mdbook-i18n-helpers

> **日期**: 2026-07-09
> **范围**: 评估是否将 `mdbook-i18n-helpers` 引入 `e:\_src\rust-lang` 以替代或补充现有的 `**EN**` / `**Summary**` 元数据方案。
> **当前 mdbook 版本**: `v0.4.52`（`book.toml` 已确认）

---

## 1. 研究现状（截至 2026-07-09）

### 1.1 `mdbook-i18n-helpers` 版本与兼容性

| 版本 | 发布时间 | mdbook 依赖 | 备注 |
|---|---|---|---|
| `0.3.6` | 2025-02-25 | `0.4.44` | 当前最新稳定版，**与项目 mdbook 0.4.52 兼容**。 |
| `0.3.5` | 2024-07-23 | — | 引入 `granularity` 优化，减少 PO 文件 churn。 |
| `0.4.0`（尚未发布稳定） | 2025-11 前后 | `0.5.x` | 上游正在适配 mdbook 0.5，但 **mdbook 0.5.0-alpha 与 0.3.6 不兼容**（issue #2835）。 |

- 核心工具：`mdbook-xgettext`（提取 `.pot`）、`mdbook-gettext`（渲染翻译）、`mdbook-i18n-normalize`（PO 格式迁移）、`i18n-report`（翻译进度报告）。
- 典型用户：Comprehensive Rust 🦀、Rust By Example、Cairo Book、Veryl HDL Book 等。

### 1.2 本项目当前方案

- 每个 `concept/` 权威页在标题后包含：
  - `**EN**`: 英文标题
  - `**Summary**`: 英文摘要（1-3 句）
- 正文为中文，**不维护完整英文翻译**，也不使用 PO/MO 文件。
- `book.toml` 中 `language = "zh"`，无多语言输出配置。

---

## 2. 方案对比

| 维度 | 当前 `**EN**` / `**Summary**` | `mdbook-i18n-helpers` gettext 方案 |
|---|---|---|
| **维护粒度** | 每页标题 + 摘要，人工维护 | 段落/句子级，自动生成 POT/PO |
| **翻译完整性** | 仅索引级英文，正文不翻译 | 可生成完整多语言书 |
| **工具链复杂度** | 低（纯 Markdown 元数据） | 中/高（gettext、PO、渲染器配置） |
| **译者协作** | 不面向专业译者 | 可直接对接 Crowdin / Transifex / PO 工作流 |
| **版本漂移风险** | 低（只有摘要） | 中（PO 文件与正文行号绑定，需定期同步） |
| **上游兼容性** | 与 mdbook 解耦 | 0.3.6 适配 0.4.x；0.5 切换需等待上游 0.4.0 |
| **迁移成本** | 已在位 | 高：400+ 概念页需提取、审校、建 PO 仓库 |
| **搜索/SEO** | mdbook 原生搜索只索引中文 | 多语言构建后可分语言索引 |
| **代码示例翻译** | 不翻译 | 默认只提取字符串字面量与注释，较合理 |

## 3. 优点与缺点

### 3.1 优点（Pros）

1. **行业标准工作流**：基于 GNU gettext，可被专业本地化平台消费。
2. **翻译进度可量化**：`i18n-report` 可生成 HTML 进度看板，便于治理。
3. **增量引入**：可先选一个子目录（如 `concept/01_foundation`）做试点。
4. **格式解耦**：现代版提取器会去掉 Markdown 语法，译者只关注纯文本。
5. **代码块智能处理**：默认只提取代码中的字符串/注释，减少噪音。

### 3.2 缺点（Cons）

1. **非官方项目**：Google 维护，但明确声明 “not an officially supported Google product”，长期维护依赖社区。
2. **mdbook 0.5 切换风险**：当 mdbook 升级到 0.5 时，0.3.6 将不可用，需等待并测试 `mdbook-i18n-helpers 0.4.0`。
3. **gettext 学习曲线**：团队需要理解 POT/PO/MO、`msgmerge`、复数形式等概念。
4. **PO 文件 churn**：行号变化会导致大量 diff，需设置 `granularity` 并配合 `mdbook-i18n-normalize`。
5. **与现有 EN/Summary 重复**：若全文翻译，现有的 `**Summary**` 会成为冗余或需要清理。
6. **构建流程变重**：CI 需要安装额外二进制、管理翻译文件、生成多语言输出。

---

## 4. 建议（Recommendation）

**阶段化策略：保持当前方案为主，2026 Q4 进行可控试点，暂不全面迁移。**

### 4.1 立即执行（2026 Q3-Q4）

- **不引入 `mdbook-i18n-helpers` 作为默认构建依赖**。
- **继续强制 `**EN**` / `**Summary**` 元数据**：它满足“可搜索、可索引、可快速理解”的最小 i18n 需求，且成本极低。
- 将 EN/Summary 质量纳入 `kb_auditor.py` 检查：确保新增/修改的 `concept/` 文件都有这两项。

### 4.2 试点计划（2026 Q4 — 2027 Q1）

- 选择 `concept/00_meta` 或 `concept/01_foundation/00_start` 作为 gettext 试点。
- 目标：验证从现有 Markdown 生成 POT、维护一份 `en.po`、并渲染英文版 book 的完整流程。
- 成功后，再评估是否推广到 `concept/` 全量。

### 4.3 迁移触发条件

只有在以下任一条件成立时，才启动全面迁移：

1. 项目明确承诺产出完整英文版书籍；
2. 有稳定译者/社区愿意维护 PO 文件；
3. `mdbook-i18n-helpers 0.4.0` 稳定发布并兼容 mdbook 0.5+，且经过本项目测试。

---

## 5. 迁移成本估算（若未来决定迁移）

| 工作项 | 估算 |
|---|---|
| 工具链与 CI 改造 | 0.5–1 人周 |
| 为 `concept/` 全量生成初始 POT | 1–2 天 |
| 翻译/审校 400+ 页的 EN 摘要到完整英文 | **20–40 人周**（取决于审校深度） |
| 建立 PO 更新与同步流程 | 1–2 人周 |
| 修复链接、代码示例、表格在多语言下的显示 | 1–2 人周 |
| 翻译进度报告与治理机制 | 0.5 人周 |
| **总计** | **约 25–50 人周**，且需持续维护成本 |

---

## 6. 结论

- **推荐保持现有 `**EN**` / `**Summary**` 方案作为 2026 Q4 的默认 i18n 实践。**
- 可在一个小目录试点 `mdbook-i18n-helpers`，以获取实际迁移经验，但不应在生产构建中强制启用。
- 全面迁移至 gettext 只有在资源充足且明确需要完整英文书时才值得投入。
